1. Что такое Spring Security

Spring Security — это мощный и настраиваемый фреймворк аутентификации и контроля доступа для приложений Java. Он обеспечивает комплексную защиту для приложений Spring.

Ключевые возможности: аутентификация, авторизация, защита от атак (CSRF, XSS, session fixation), интеграция с OAuth2, SAML, LDAP, JWT и многое другое.
graph TB Client[Client] --> FilterChain[Security Filter Chain] FilterChain --> Auth[Authentication Manager] FilterChain --> Authz[Access Decision Manager] Auth --> Provider[Authentication Provider] Provider --> DB[(Database)] Authz --> Voting[Access Decision Voting]

Проект использует H2 в качестве in-memory базы данных, схема создаётся автоматически через JPA (ddl-auto=create-drop), начальные данные загружаются через DemoDataInitializer.java.

2. Идентификация, Аутентификация и Авторизация

Три ключевых процесса безопасности, которые часто путают, но выполняют принципиально разные функции. Различие между аутентификацией и авторизацией возвращается эндпоинтом /api/security/auth-vs-authorization и проверяется тестом Section3_AuthVsAuthorizationTest.java.

Идентификация (Identification) — «Кто ты?»

Идентификация — процесс заявления о своей личности. Пользователь называет себя, но ещё ничего не доказывает. Система ищет предъявленный идентификатор в своей базе данных.

Типичные идентификаторы:
  • Логин (username)
  • Email-адрес
  • Номер телефона
  • ID карты / badge
  • Номер паспорта
  • Отпечаток пальца (при поиске по базе — это тоже идентификация)
⚠️ На этапе идентификации доверия ещё нет! Любой может заявить: «Я — admin». Это не значит, что система ему поверит. 
// Пользователь вводит логин — это ИДЕНТИФИКАЦИЯ
POST /api/auth/login
{
    "username": "ivan_petrov",   // ← Идентификация: "Я — ivan_petrov"
    "password": "***"            // ← Это уже для аутентификации (следующий шаг)
}

// Сервер: "Ок, я нашёл такого пользователя. Теперь докажи, что это ты."

Идентификация в Spring Security

В Spring Security идентификация происходит внутри UserDetailsService.loadUserByUsername() — система ищет пользователя по предъявленному идентификатору: 

@Service
public class CustomUserDetailsService implements UserDetailsService {

    @Override
    public UserDetails loadUserByUsername(String username)
            throws UsernameNotFoundException {

        // ← Это ИДЕНТИФИКАЦИЯ: поиск пользователя по заявленному имени
        User user = userRepository.findByUsername(username)
            .orElseThrow(() -> new UsernameNotFoundException(
                "Пользователь не найден: " + username  // Идентификация не прошла
            ));

        return buildUserDetails(user);
        // Далее AuthenticationProvider проверит пароль — это уже АУТЕНТИФИКАЦИЯ
    }
}

Аутентификация (Authentication) — «Докажи, что ты — тот, за кого себя выдаёшь!»

Аутентификация — процесс подтверждения заявленной личности. Пользователь предоставляет доказательства (credentials), что он действительно тот, кем представился на этапе идентификации.

📱 ВладениеЧто-то, что вы имеетеТелефон (SMS/TOTP), смарт-карта, USB-токен
🧬 СвойствоЧто-то, что вы естьОтпечаток пальца, распознавание лица, сетчатка

Аутентификация в Spring Security

// Полный путь: Идентификация → Аутентификация
public Authentication authenticate(Authentication authentication) {

    String username = authentication.getName();
    String password = authentication.getCredentials().toString();

    // 1. ИДЕНТИФИКАЦИЯ — поиск пользователя
    UserDetails user = userDetailsService.loadUserByUsername(username);
    //   └── Если не найден → UsernameNotFoundException

    // 2. АУТЕНТИФИКАЦИЯ — проверка пароля
    if (!passwordEncoder.matches(password, user.getPassword())) {
        throw new BadCredentialsException("Неверный пароль");
        //   └── Идентификация прошла, но аутентификация НЕТ
    }

    // ✅ Оба этапа пройдены — личность подтверждена
    return new UsernamePasswordAuthenticationToken(
        user, null, user.getAuthorities()
    );
}

Авторизация (Authorization) — «Что тебе разрешено делать?»

Авторизация — процесс определения прав доступа уже подтверждённого пользователя. Система проверяет, какие ресурсы доступны и какие действия разрешены.

Авторизация определяет:
  • Какие ресурсы доступны?
  • Какие действия разрешены (чтение, запись, удаление)?
  • Какой уровень доступа у пользователя?

Авторизация в Spring Security

@GetMapping("/api/users")
@PreAuthorize("hasRole('ADMIN')")  // ← АВТОРИЗАЦИЯ: только админы
public List<UserDto> getAllUsers() { ... }

@GetMapping("/api/users/{id}")
@PreAuthorize("hasRole('ADMIN') or #id == authentication.principal.id")
public UserDto getUserById(@PathVariable Long id) { ... }
// ↑ АВТОРИЗАЦИЯ: админ ИЛИ владелец профиля

Особый случай: биометрия

Биометрия может выступать в двух ролях:

Как идентификация — система ищет лицо в базе всех сотрудников (поиск «один ко многим»: «Чьё это лицо?»)

Как аутентификация — система сравнивает отпечаток с конкретным пользователем, который уже назвал своё имя (сравнение «один к одному»: «Это действительно Иван?»)

Типичные ошибки в понимании

❌ Неправильно✅ Правильно
«Логин и пароль — это аутентификация» Логин — идентификация, пароль — аутентификация
«401 и 403 — одно и то же» 401 — не аутентифицирован, 403 — не авторизован
«Авторизация = вход в систему» Вход = идентификация + аутентификация. Авторизация = проверка прав
«Идентификация не нужна, хватит аутентификации» Без идентификации система не знает, чьи учётные данные проверять
«Биометрия — это только аутентификация» Биометрия может быть и идентификацией (поиск по базе лиц), и аутентификацией (подтверждение конкретного пользователя)

Резюме

Строгая последовательность:

Идентификация → Аутентификация → Авторизация

Нельзя аутентифицировать того, кто не идентифицирован.
Нельзя авторизовать того, кто не аутентифицирован.

3. Архитектура Spring Security

Spring Security построен на цепочке фильтров (Security Filter Chain), через которую проходит каждый HTTP-запрос перед тем, как достичь вашего контроллера.

Ключевые компоненты

  • SecurityContext — хранит информацию о текущем аутентифицированном пользователе (Principal).
  • SecurityContextHolder — статический класс для доступа к SecurityContext текущего потока.
  • Authentication — представляет токен аутентификации (credentials, principal, authorities).
  • AuthenticationManager — интерфейс для обработки аутентификации.
  • UserDetailsService — загрузка пользовательских данных по username.
  • PasswordEncoder — шифрование и проверка паролей (BCrypt, Argon2, etc.).

Диаграмма архитектуры

4. Security Filter Chain

Цепочка фильтров (Filter Chain) — это ядро Spring Security. Каждый фильтр выполняет определённую задачу: проверку CSRF, базовую аутентификацию, обработку формы логина и т.д.

Фильтры выполняются в определённом порядке. Если фильтр не может обработать запрос, он передаёт управление следующему фильтру в цепочке.

Основные фильтры в цепочке

ПорядокФильтрОписание
1ChannelProcessingFilterПроверяет, должен ли запрос быть по HTTPS
3SecurityContextPersistenceFilterЗагружает/сохраняет SecurityContext
4HeaderWriterFilterДобавляет security headers
5CorsFilterОбрабатывает CORS запросы
6CsrfFilterЗащита от CSRF атак
8UsernamePasswordAuthenticationFilterОбрабатывает форму логина
9BasicAuthenticationFilterОбрабатывает HTTP Basic
14ExceptionTranslationFilterПреобразует исключения в HTTP ответы
15FilterSecurityInterceptorАвторизует запросы
В проекте SecurityConfiguration.java настраивает SecurityFilterChain, отключает CSRF для REST API и определяет правила доступа к endpoint'ам.

5. Конфигурация безопасности

Основная конфигурация безопасности определена в классе SecurityConfiguration.java. Она настраивает SecurityFilterChain, PasswordEncoder и AuthenticationManager.

Основные настройки конфигурации

@Configuration
@EnableWebSecurity
@EnableMethodSecurity
public class SecurityConfiguration {

  @Bean
  public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
    http
        .csrf(AbstractHttpConfigurer::disable)
        .authorizeHttpRequests(auth -> auth
            .requestMatchers("/api/auth/**", "/api/security/**").permitAll()
            .anyRequest().authenticated()
        )
        .httpBasic(Customizer.withDefaults())
        .formLogin(form -> form
            .loginPage("/login")
            .permitAll()
        );

    return http.build();
  }

  @Bean
  public PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder();
  }
}

Пояснение настроек

  • @EnableWebSecurity — включает Spring Security веб-безопасность.
  • @EnableMethodSecurity — включает аннотации для безопасности на уровне методов (@PreAuthorize, @Secured).
  • csrf().disable() — отключает CSRF (для REST API с JWT).
  • authorizeHttpRequests() — определяет правила доступа к endpoint'ам.
  • httpBasic() — включает HTTP Basic аутентификацию.
  • formLogin() — настраивает форму логина.
  • passwordEncoder() — BCrypt для хеширования паролей.
Важно: Для production приложений не отключайте CSRF без веских причин. Используйте JWT или другой stateless механизм аутентификации вместе с CSRF защитой.

6. Password Encoding

Пароли никогда не должны храниться в открытом виде. Spring Security предоставляет интерфейс PasswordEncoder для шифрования паролей. Проект использует BCrypt — один из самых популярных и безопасных алгоритмов.

BCrypt Password Encoder

BCrypt автоматически генерирует соль (salt) при каждом хешировании и включает её в результат. Это делает каждый хеш уникальным, даже для одинаковых паролей. 

PasswordEncoder encoder = new BCryptPasswordEncoder();

// Хеширование пароля
String encodedPassword = encoder.encode("password123");
// Результат: $2a$10$N9qo8uLOickgx2ZMRZoMy...

// Проверка пароля
boolean matches = encoder.matches("password123", encodedPassword);
Преимущества BCrypt:
- Встроенная соль (не нужно хранить отдельно)
- Активная работа по замедлению (сложность можно настраивать)
- Широкая поддержка и проверка временем

Настройка strength (сильны)

По умолчанию BCrypt использует 10 rounds (итераций). Можно изменить силу: 

@Bean
public PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder(12); // Более безопасно, но медленнее
}

7. Пользователи, роли и разрешения (RBAC)

Проект реализует RBAC (Role-Based Access Control) с тремя уровнями: Users (пользователи), Roles (роли) и Permissions (разрешения).

Модель данных

classDiagram class User { Long id String username String password String email Boolean enabled Set~Role~ roles } class Role { Long id String name String description Set~Permission~ permissions Set~User~ users } class Permission { Long id String name String description Set~Role~ roles } User "many" -- "many" Role : has Role "many" -- "many" Permission : has

Сущности в проекте

  • User — учётная запись пользователя с полями username, password, email, enabled, roles.
  • Role — роль (USER, MODERATOR, ADMIN) с набором разрешений.
  • Permission — детальное разрешение (USER_READ, USER_WRITE, ROLE_READ, etc.).

Иерархия прав в демо-данных

РольРазрешения
USERUSER_READ
MODERATORUSER_READ, USER_WRITE, ROLE_READ
ADMINВсе разрешения
Демо-пользователи создаются в DemoDataInitializer.java. Пароли хешируются через BCrypt перед сохранением в базу.

8. Method Security

Spring Security обеспечивает безопасность на уровне методов с помощью аннотаций. Это позволяет контролировать доступ к конкретным методам сервиса или контроллера.

9. Security Events

Spring Security предоставляет события, которые происходят при различных действиях безопасности. Проект использует SecurityEventsCollector.java для сбора таких событий.

Типы событий

  • AUTHENTICATION_SUCCESS — успешная аутентификация пользователя.
  • AUTHENTICATION_FAILED — неудачная попытка входа.
  • USER_CREATED — создание нового пользователя.
  • USER_DELETED — удаление пользователя.
  • USER_ENABLED / USER_DISABLED — включение/отключение учётной записи.
  • ROLE_ASSIGNED / ROLE_REMOVED — назначение/удаление роли.
  • PASSWORD_CHANGED — изменение пароля.

Пример события

eventsCollector.recordEvent("AUTHENTICATION_SUCCESS", username,
    "User logged in successfully");
В продакшене эти события могут быть отправлены в систему логирования, мониторинга, аудита или использованы для уведомлений пользователей о подозрительной активности.

13. Быстрый старт (curl)

Примеры запросов для быстрого тестирования API после запуска приложения:

Получить концепции безопасности

curl http://localhost:8892/api/security/concepts | jq

Аутентификация

curl -X POST http://localhost:8892/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username": "admin", "password": "admin123"}' | jq

Получить всех пользователей (HTTP Basic auth)

curl -u admin:admin123 \
  http://localhost:8892/api/users | jq

Создать пользователя

curl -X POST http://localhost:8892/api/users \
  -H "Content-Type: application/json" \
  -u admin:admin123 \
  -d '{
    "username": "testuser",
    "password": "test123",
    "email": "test@example.com",
    "firstName": "Test",
    "lastName": "User"
  }' | jq

Назначить роль пользователю

curl -X POST http://localhost:8892/api/users/2/roles/2 \
  -H "Content-Type: application/json" \
  -u admin:admin123 | jq

Отключить пользователя

curl -X PUT http://localhost:8892/api/users/2/disable \
  -H "Content-Type: application/json" \
  -u admin:admin123 | jq

Получить все роли

curl http://localhost:8892/api/roles | jq

Получить все разрешения

curl http://localhost:8892/api/permissions | jq

Получить события безопасности

curl http://localhost:8892/api/security/events | jq

14. Тесты

Каждый ключевой аспект покрыт интеграционными тестами, разбитыми по секциям:

  • Section1_SecurityConceptsTest.java — проверяет ключевые концепции.
  • Section2_FilterChainTest.java — проверяет цепочку фильтров.
  • Section3_AuthVsAuthorizationTest.java — сравнение аутентификации и авторизации.
  • Section4_MethodSecurityTest.java — аннотации метод-безопасности.
  • Section5_SecurityConfigTest.java — настройки конфигурации.
  • Section6_GlossaryTest.java — наличие основных терминов.
  • Section7_EndpointsTest.java — проверка REST endpoint'ов.
  • Section8_ArchitectureTest.java — обзор архитектуры.
Тесты можно запустить командой mvn clean test в модуле.

Вывод

Spring Security предоставляет комплексное решение для аутентификации и авторизации в Spring-приложениях. Этот проект демонстрирует все ключевые возможности: RBAC с пользователями, ролями и разрешениями, method security, password encoding, security events, конфигурацию цепочки фильтров и полноценный REST API.
