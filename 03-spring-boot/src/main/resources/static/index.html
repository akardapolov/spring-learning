1. Запуск приложения

Аннотация @SpringBootApplication — это мета-аннотация, которая объединяет три ключевые аннотации Spring Boot: @Configuration, @EnableAutoConfiguration и @ComponentScan. 

@SpringBootApplication
@ConfigurationPropertiesScan
public class SpringBootDemoApplication {

    public static void main(String[] args) {
        SpringApplication.run(SpringBootDemoApplication.class, args);
    }
}

Процесс запуска

1. main() — точка входа, вызывается SpringApplication.run()
2. Создание ApplicationContext — Spring загружает контекст приложения
3. Auto-Configuration — анализ classpath и автоматическая настройка
4. Bean Creation — сканирование компонентов и создание бинов
5. Embedded Server — запуск Tomcat на указанном порту
graph TB MAIN["main - точка входа"] --> APP_CTX["ApplicationContext"] APP_CTX --> AUTO["Auto-Configuration: анализ classpath"] AUTO --> BEANS["Bean Creation: сканирование Component"] BEANS --> TOMCAT["Embedded Tomcat: запуск сервера"]

Варианты запуска

Вариант 1 — через Maven

mvn clean spring-boot:run -pl 03-spring-boot

Вариант 2 — через собранный JAR

mvn clean package -pl 03-spring-boot
java -jar 03-spring-boot/target/03-spring-boot-1.0.0.jar
Ключевой принцип: Spring Boot берёт на себя всю конфигурацию. Вам остаётся писать бизнес-логику.

2. Configuration Properties

@ConfigurationProperties обеспечивает типобезопасную привязку конфигурации из YAML/properties-файлов в Java-объекты с поддержкой валидации и вложенных свойств.

Ключевая идея: вместо разбросанных @Value("${app.name}") вы создаёте один класс-контейнер, где все свойства собраны и типизированы. Компилятор и IDE проверяют корректность ещё на этапе компиляции.

Конфигурация в YAML

app:
  name: Spring Boot Demo Application
  version: 1.0.0
  message:
    welcome: "Welcome to Spring Boot Demo!"
    greeting: "Hello from Spring Boot!"
  features:
    cache-enabled: true
    metrics-enabled: true
    logging-enabled: true
    debug-mode: false

Java-класс привязки

@Data
@Validated
@ConfigurationProperties(prefix = "app")
public class AppProperties {

    @NotBlank(message = "Application name cannot be blank")
    private String name;

    @NotBlank(message = "Application version cannot be blank")
    private String version;

    @NotNull
    private String environment;

    private Message message = new Message();
    private Features features = new Features();

    @Data
    public static class Message {
        @NotBlank
        private String welcome;

        @NotBlank
        private String greeting;
    }

    @Data
    public static class Features {
        private boolean cacheEnabled = true;
        private boolean metricsEnabled = true;
        private boolean loggingEnabled = true;
        private boolean debugMode = false;
        private boolean detailedLogging = false;
    }
}

Использование профилей в коде

@Service
public class ProfileService {

    @Autowired
    private Environment environment;

    public String getCurrentProfile() {
        String[] profiles = environment.getActiveProfiles();
        return profiles.length > 0 ? profiles[0] : "default";
    }

    public boolean isProfileActive(String profile) {
        return environment.acceptsProfiles(Profiles.of(profile));
    }

    public List getActiveProfiles() {
        return Arrays.asList(environment.getActiveProfiles());
    }

    public String getProfileSpecificMessage() {
        if (isProfileActive("dev")) {
            return "Running in development mode with debug enabled";
        } else if (isProfileActive("prod")) {
            return "Running in production mode with optimizations";
        } else if (isProfileActive("test")) {
            return "Running in test mode";
        }
        return "Running in default mode";
    }
}

4. Инспекция бинов

ApplicationContext — центральный интерфейс Spring, позволяющий получать информацию о зарегистрированных бинах: их имена, типы, области видимости и зависимости. 

@Service
public class BeanService {

    @Autowired
    private ApplicationContext applicationContext;

    public List<BeanInfo> getAllBeans() {
        String[] beanNames = applicationContext.getBeanDefinitionNames();
        List<BeanInfo> beans = new ArrayList<>();

        for (String beanName : beanNames) {
            try {
                Object bean = applicationContext.getBean(beanName);
                String scope = "singleton";
                if (applicationContext instanceof ConfigurableApplicationContext cac) {
                    var definition = cac.getBeanFactory().getBeanDefinition(beanName);
                    scope = definition.getScope();
                    if (scope == null || scope.isEmpty()) {
                        scope = "singleton";
                    }
                }
                beans.add(BeanInfo.builder()
                        .name(beanName)
                        .type(bean.getClass().getName())
                        .scope(scope)
                        .isSingleton(applicationContext.isSingleton(beanName))
                        .isPrototype(applicationContext.isPrototype(beanName))
                        .build());
            } catch (Exception e) {
                log.debug("Could not inspect bean: {}", beanName);
            }
        }

        return beans.stream()
                .sorted(Comparator.comparing(BeanInfo::getName))
                .toList();
    }

    public long getBeanCount() {
        return applicationContext.getBeanDefinitionNames().length;
    }
}

5. Условные бины

Условные аннотации позволяют создавать бины только при выполнении определённых условий. Это основа auto-конфигурации Spring Boot.

Ключевая идея: Spring Boot проверяет условие перед созданием бина. Если условие не выполнено — бин просто не создаётся, без ошибок. Это позволяет переопределять авто-конфигурацию, просто определив свой бин.

Ключевые аннотации

АннотацияУсловиеПример использования
@ConditionalOnPropertyСвойство имеет определённое значениеВключение фичи через конфиг
@ConditionalOnMissingBeanНет бина данного типаДефолтная реализация
@ConditionalOnClassКласс есть в classpathИнтеграция с библиотекой
@ConditionalOnExpressionSpEL-выражение истинноСложные комбинированные условия

Пример: условный DebugService

@Bean
@ConditionalOnProperty(
    prefix = "app.features",
    name = "debug-mode",
    havingValue = "true"
)
public DebugService debugService(AppProperties appProperties) {
    log.info("Creating DebugService - debug mode is enabled");
    return new DebugService(appProperties);
}

Пример: DefaultCacheService (@ConditionalOnMissingBean)

@Bean
@ConditionalOnMissingBean(name = "customCacheService")
public DefaultCacheService defaultCacheService() {
    log.info("Creating DefaultCacheService - no custom cache service found");
    return new DefaultCacheService();
}

Использование в ConditionalBeanService

@Service
public class ConditionalBeanService {

    private final AppProperties appProperties;

    public String getConditionalMessage() {
        if (appProperties.getFeatures().isDebugMode()) {
            return "Debug mode is enabled! Conditional bean is active.";
        } else {
            return "This conditional bean was created (without debug mode requirement).";
        }
    }

    public boolean isFeatureEnabled(String feature) {
        return switch (feature.toLowerCase()) {
            case "cache" -> appProperties.getFeatures().isCacheEnabled();
            case "metrics" -> appProperties.getFeatures().isMetricsEnabled();
            case "logging" -> appProperties.getFeatures().isLoggingEnabled();
            case "debug" -> appProperties.getFeatures().isDebugMode();
            default -> false;
        };
    }
}

6. Внедрение зависимостей (DI)

Spring управляет созданием и связыванием объектов. Существует несколько способов внедрения зависимостей, но есть один рекомендуемый — конструкторная инъекция.

6.1 Конструкторная инъекция (рекомендуется)

@Service
@RequiredArgsConstructor  // Lombok генерирует конструктор
public class AppInfoService {

    private final AppProperties appProperties;  // final — обязательно инициализировать

    @Autowired
    private Environment environment;

    public ApiModels.AppInfo getAppInfo() {
        return ApiModels.AppInfo.builder()
                .name(appProperties.getName())
                .version(appProperties.getVersion())
                .environment(appProperties.getEnvironment())
                .features(getEnabledFeatures())
                .build();
    }

    public String getWelcomeMessage() {
        return appProperties.getMessage().getWelcome();
    }

    public String getGreetingMessage() {
        return appProperties.getMessage().getGreeting();
    }
}

6.2 Конфликт типов: @Qualifier и @Primary

Когда в контексте несколько бинов одного типа, Spring не знает, какой из них использовать. Решение — @Primary (для дефолтного) или @Qualifier (для явного выбора). 

// Когда есть несколько бинов одного типа:

@Bean
@Primary  // Этот бин будет выбран по умолчанию
public CacheService primaryCache() {
    return new DefaultCacheService();
}

@Bean
public CacheService redisCache() {
    return new RedisCacheService();
}

// Используйте @Qualifier для явного выбора:
public MyService(@Qualifier("redisCache") CacheService cache) {
    this.cache = cache;
}
В проекте используется @Qualifier в тестах для демонстрации явного выбора бина при инъекции.

7. Environment и окружение

Интерфейс Environment предоставляет доступ ко всем свойствам приложения и позволяет читать конфигурацию из различных источников с учётом их приоритета.

Приоритет источников конфигурации

Свойства из источника с более высоким приоритетом перекрывают значения из источников с более низким:

1 (макс. приоритет). Аргументы командной строки — --server.port=9090
2. System properties — -Dapp.name=Test
3. OS environment variables — переменные окружения ОС
4. application-{profile}.yaml — профиль-специфичные файлы
5. application.yaml — основной файл конфигурации
6. @PropertySource — дополнительные источники
7 (мин. приоритет). Значения по умолчанию — @Value("${key:default}")

Использование Environment

@Service
public class ConfigurationService {

    @Autowired
    private Environment environment;

    @Autowired
    private ConfigurableEnvironment configurableEnvironment;

    public String getProperty(String key) {
        String value = environment.getProperty(key);
        if (value == null) {
            throw new ConfigurationException(key, "Property not found");
        }
        return value;
    }

    public String getProperty(String key, String defaultValue) {
        return environment.getProperty(key, defaultValue);
    }

    public List<String> getConfigurationSources() {
        List<String> sources = new ArrayList<>();
        for (PropertySource<?> source : configurableEnvironment.getPropertySources()) {
            sources.add(source.getName());
        }
        return sources;
    }

    public List<String> getActiveProfiles() {
        return List.of(environment.getActiveProfiles());
    }
}

8. REST-контроллеры

REST-контроллеры обрабатывают HTTP-запросы и возвращают данные в формате JSON. В проекте используется стандартный формат ответа ApiResponse<T>. 

@RestController
@RequestMapping("/api/demo")
@RequiredArgsConstructor
public class DemoController {

    private final AppInfoService appInfoService;
    private final BeanService beanService;
    private final ConfigurationService configurationService;
    private final ConditionalBeanService conditionalBeanService;
    private final ProfileService profileService;

    // GET без параметров
    @GetMapping("/app-info")
    public ResponseEntity<ApiResponse<AppInfo>> getAppInfo() {
        return ResponseEntity.ok(ApiResponse.success(appInfoService.getAppInfo()));
    }

    // GET с параметром запроса
    @GetMapping("/greeting")
    public ResponseEntity<ApiResponse<String>> getGreeting(
            @RequestParam(defaultValue = "Guest") String name) {
        String greeting = appInfoService.getGreetingMessage() + ", " + name + "!";
        return ResponseEntity.ok(ApiResponse.success(greeting));
    }

    // GET с переменной пути
    @GetMapping("/beans/{beanName}")
    public ResponseEntity<ApiResponse<BeanInfo>> getBeanInfo(
            @PathVariable String beanName) {
        return ResponseEntity.ok(ApiResponse.success(beanService.getBeanInfo(beanName)));
    }
}

Стандартный формат ответа ApiResponse

@Data
@Builder
@JsonInclude(JsonInclude.Include.NON_NULL)
public static class ApiResponse<T> {
    private boolean success;
    private String message;
    private T data;
    private List<String> errors;
    private LocalDateTime timestamp;

    public static <T> ApiResponse<T> success(T data) {
        return ApiResponse.<T>builder()
                .success(true)
                .data(data)
                .timestamp(LocalDateTime.now())
                .build();
    }

    public static <T> ApiResponse<T> error(String message) {
        return ApiResponse.<T>builder()
                .success(false)
                .message(message)
                .timestamp(LocalDateTime.now())
                .build();
    }
}

9. Обработка ошибок

Аннотация @RestControllerAdvice позволяет централизованно обрабатывать исключения во всех контроллерах, возвращая единообразные ответы об ошибках.

Иерархия исключений

graph TB RE["RuntimeException"] --> APP["AppException: APP_ERROR"] APP --> BEAN["BeanNotFoundException: BEAN_NOT_FOUND"] APP --> CONFIG["ConfigurationException: CONFIG_ERROR"]

GlobalExceptionHandler

@Slf4j
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(AppException.class)
    public ResponseEntity<ApiResponse<Void>> handleAppException(AppException ex) {
        log.warn("Application exception: {}", ex.getMessage());
        return ResponseEntity
                .status(HttpStatus.BAD_REQUEST)
                .body(ApiResponse.error(ex.getMessage()));
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<ApiResponse<Map<String, String>>> handleValidation(
            MethodArgumentNotValidException ex) {
        Map<String, String> errors = new HashMap<>();
        ex.getBindingResult().getAllErrors().forEach(error -> {
            String fieldName = ((FieldError) error).getField();
            errors.put(fieldName, error.getDefaultMessage());
        });
        return ResponseEntity
                .status(HttpStatus.BAD_REQUEST)
                .body(ApiResponse.error("Validation failed", List.of(errors.toString())));
    }

    @ExceptionHandler(IllegalArgumentException.class)
    public ResponseEntity<ApiResponse<Void>> handleIllegalArgument(IllegalArgumentException ex) {
        return ResponseEntity
                .status(HttpStatus.BAD_REQUEST)
                .body(ApiResponse.error(ex.getMessage()));
    }

    @ExceptionHandler(Exception.class)
    public ResponseEntity<ApiResponse<Void>> handleGeneric(Exception ex) {
        log.error("Unexpected error", ex);
        return ResponseEntity
                .status(HttpStatus.INTERNAL_SERVER_ERROR)
                .body(ApiResponse.error("An unexpected error occurred"));
    }
}

Кастомные исключения

public class AppException extends RuntimeException {
    private final String code;

    public AppException(String code, String message) {
        super(message);
        this.code = code;
    }

    public String getCode() { return code; }
}

public class BeanNotFoundException extends AppException {
    public BeanNotFoundException(String beanName) {
        super("BEAN_NOT_FOUND", "Bean '" + beanName + "' not found");
    }
}

public class ConfigurationException extends AppException {
    public ConfigurationException(String key, String reason) {
        super("CONFIG_ERROR", "Configuration error for key '" + key + "': " + reason);
    }
}
Порядок обработчиков важен: Spring выбирает наиболее специфичный @ExceptionHandler. Общий обработчик Exception ставится последним как fallback. Более специфичные (например, AppException) перехватываются первыми.

Кастомные метрики

@Configuration
public class CustomMetricsConfig {

    @Autowired
    private MeterRegistry meterRegistry;

    private Counter requestCounter;

    @PostConstruct
    public void initMetrics() {
        requestCounter = Counter.builder("api.requests.total")
                .description("Total number of API requests")
                .tag("type", "demo")
                .register(meterRegistry);

        meterRegistry.gauge("app.beans.loaded", 0);
    }

    public void recordRequest() {
        requestCounter.increment();
    }
}
Безопасность: в продакшене ограничьте exposure endpoints через management.endpoints.web.exposure.include и настройте авторизацию. Не открывайте /actuator/env и /actuator/configprops публично — они могут содержать чувствительные данные.

11. Auto-Configuration

Spring Boot автоматически конфигурирует приложение на основе зависимостей в classpath. Это делает настройку минимальной — достаточно добавить нужный starter в pom.xml.

graph LR CP["Classpath: зависимости в pom.xml"] --> AC["Auto-Configuration"] AC --> COND["Conditions: ConditionalOn"] COND --> BEANS["Beans: создаются автоматически"] BEANS --> APP["Готовое приложение"]

Просмотр отчёта авто-конфигурации

mvn spring-boot:run -pl 03-spring-boot -Ddebug

В debug-режиме Spring Boot выведет в лог полный отчёт: какие авто-конфигурации были применены, а какие — пропущены и почему.

Пример REST-теста

@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)
class Section8_RestControllerTest {

    @Autowired
    private TestRestTemplate restTemplate;

    @LocalServerPort
    private int port;

    @Test
    void getAppInfoReturnsValidData() {
        ResponseEntity<Map> response = restTemplate.getForEntity(
            "http://localhost:" + port + "/api/demo/app-info", Map.class
        );

        assertThat(response.getStatusCode().is2xxSuccessful()).isTrue();
        assertThat(response.getBody()).isNotNull();
        assertThat(response.getBody().get("success")).isEqualTo(true);

        Map<String, Object> data = (Map<String, Object>) response.getBody().get("data");
        assertThat(data.get("name")).isNotNull();
        assertThat(data.get("version")).isNotNull();
    }
}

16. Best Practices

Делать ✅

#ПрактикаПочему
1Использовать конструкторную инъекциюfinal-поля, тестируемость, читаемость
2Делать бины stateless (singleton)Потокобезопасность, производительность
3Использовать @ConfigurationPropertiesТипобезопасность, валидация
4Разделять конфигурацию по профилямЧистота окружений
5Писать интеграционные тестыДоверие к коду, отсутствие регрессий
6Использовать @RestControllerAdviceЦентрализованная обработка ошибок
7Включать Actuator в productionМониторинг, health checks
8Валидировать входные данныеЗащита от некорректных данных

Избегать ❌

#АнтипаттернПочему
1Полевой инъекции (@Autowired на поле)Нельзя final, сложнее тестировать
2Циклических зависимостейНарушение принципа единственной ответственности
3Хранения состояния в singleton-бинахГонки данных, непредсказуемое поведение
4Отключения Actuator без авторизацииУтечка чувствительных данных
5Игнорирования исключенийСкрытые баги, сложная отладка
6Создания бинов через newНарушение DI, невозможно тестировать

Вывод

Spring Boot делает создание production-ready приложений простым и предсказуемым. Auto-configuration берёт на себя рутину, @ConfigurationProperties обеспечивает типобезопасную конфигурацию, профили позволяют гибко управлять окружениями, а Actuator предоставляет инструменты мониторинга из коробки. Этот проект демонстрирует все ключевые концепции фреймворка на практических примерах с полным покрытием тестами.
