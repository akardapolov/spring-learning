1. Что такое IoC Container

IoC (Inversion of Control) — принцип проектирования, при котором управление жизненным циклом объектов и создание их зависимостей передаётся внешнему фреймворку (контейнеру), а не самим объектам. В Spring этот контейнер называется ApplicationContext.

Ключевая идея: в традиционном подходе объект сам создаёт свои зависимости через new. При IoC контейнер читает конфигурацию (аннотации, Java-конфиг), создаёт объекты (бины) и «скручивает» (wiring) их вместе. Ваш код просто запрашивает готовый объект из контейнера.

ApplicationContext — основной интерфейс контейнера. Он расширяет BeanFactory, добавляя поддержку AOP, событий, профилей и интернационализации.

2. Dependency Injection — три способа инъекции

Dependency Injection (DI) — конкретная реализация IoC, при которой контейнер внедряет зависимости в объект. Spring поддерживает три способа инъекции.

СпособАннотацияРекомендацияПример
Конструкторная @Autowired (или без неё) ✅ Рекомендуется public Foo(Bar bar) { … }
Setter @Autowired на сеттер ⚠️ Опциональные зависимости public void setBar(Bar b) { … }
Field @Autowired на поле ❌ Не рекомендуется @Autowired private Bar bar;

передаётся через конструктор| C2[ProductRepository] end subgraph Setter["2. Setter инъекция"] S1[ProductService
@Autowired setDefaultCategory] -->|Вызывается сеттер
с @Value| S2[defaultCategory = 'Uncategorized'] end subgraph Field["3. Field инъекция (❌)"] F1[ProductService
@Autowired private ...] -->|Внедряется напрямую в поле| F2[defaultProduct] end

Конструкторная инъекция (рекомендуемый подход)

@Service
public class ProductService {

    private final ProductRepository productRepository;

    // @Autowired необязателен при единственном конструкторе (Spring 4.3+)
    public ProductService(ProductRepository productRepository) {
        this.productRepository = productRepository;
    }

    public List<Product> findAll() {
        return productRepository.findAll();
    }
}

Setter-инъекция

@Service
public class ProductService {

    private String defaultCategory;

    @Autowired
    public void setDefaultCategory(
            @Value("${app.default-category:Uncategorized}") String defaultCategory) {
        this.defaultCategory = defaultCategory;
    }
}

Field-инъекция (не рекомендуется)

@Service
public class ProductService {

    // ⚠️ Не рекомендуется: скрывает зависимости, мешает тестированию
    @Autowired
    @Qualifier("defaultProduct")
    private Product defaultProduct;
}

Почему конструкторная инъекция лучше?

  • Неизменяемость: поля можно сделать final
  • Явность: все зависимости видны в сигнатуре конструктора
  • Тестируемость: легко создать объект в тесте, передав mock
  • Fail-fast: контейнер не стартует, если зависимость отсутствует
Field injection — антипаттерн. Скрывает зависимости, делает класс неизменяемым, усложняет модульное тестирование (нужен Reflection или Spring Test Context). Используйте только в демонстрационных целях, как в этом проекте.
В проекте: ProductService.java демонстрирует все три способа одновременно — конструкторную (ProductRepository), setter (defaultCategory через @Value) и field (defaultProduct через @Autowired + @Qualifier). Тест Section2_DependencyInjectionTest.java проверяет каждый из них.

3. Bean Scope — singleton vs prototype

Scope определяет, сколько экземпляров бина создаёт контейнер и как долго они живут. По умолчанию все бины — singleton.

ScopeАннотацияКоличество экземпляровОбласть применения
singleton (по умолчанию) 1 на контейнер Все слои
prototype @Scope("prototype") Новый при каждом запросе Состояние-зависимые бины
request @RequestScope 1 на HTTP-запрос Web
session @SessionScope 1 на HTTP-сессию Web
application @ApplicationScope 1 на ServletContext Web

экземпляр #1"] S2["Запрос 2"] --> Bean1 S3["Запрос 3"] --> Bean1 end subgraph Prototype["Prototype Scope"] P1["Запрос 1"] --> BeanP1["Bean B
экземпляр #1"] P2["Запрос 2"] --> BeanP2["Bean B
экземпляр #2"] P3["Запрос 3"] --> BeanP3["Bean B
экземпляр #3"] end

Пример из проекта

// Singleton (по умолчанию) — один экземпляр на весь контейнер
@Service
public class ProductService {
    // ...
}

// Prototype — новый экземпляр при каждом вызове getBean()
@Service
@Scope("prototype")
public class PrototypeProductService {

    private final Product currentProduct;

    public PrototypeProductService() {
        this.currentProduct = new Product(
            "Prototype Product", "PROTOTYPE", BigDecimal.ONE);
    }

    public Product getCurrentProduct() {
        return currentProduct;
    }
}
@Test
void shouldShowScopeDifference() {
    // Singleton — всегда один и тот же объект
    Product p1 = context.getBean(ProductService.class).getDefaultProduct();
    Product p2 = context.getBean(ProductService.class).getDefaultProduct();
    assertThat(p1).isSameAs(p2);
    assertThat(p1.getId()).isEqualTo(p2.getId());

    // Prototype — каждый раз новый объект
    PrototypeProductService t1 = context.getBean(PrototypeProductService.class);
    PrototypeProductService t2 = context.getBean(PrototypeProductService.class);
    assertThat(t1).isNotSameAs(t2);
    assertThat(t1.getCurrentProduct().getId())
            .isNotEqualTo(t2.getCurrentProduct().getId());
}
Singleton → Prototype проблема. Если singleton-бин инжектит prototype-бин, prototype будет создан один раз (при создании singleton). Чтобы получать новый prototype каждый раз, используйте ObjectProvider<T>, Provider<T> или @Lookup.
В проекте: PrototypeProductService.java с @Scope("prototype"). Тест Section3_ScopeTest.java проверяет разницу между singleton и prototype. Метод scopeDemo() возвращает ScopeDemoResult с идентификаторами экземпляров.

4. Lifecycle бина — @PostConstruct и @PreDestroy

Каждый бин проходит через набор этапов жизненного цикла: создание → инъекция зависимостей → инициализация → использование → уничтожение.

flowchart LR A["1. Конструктор
создание объекта"] --> B["2. Инъекция
зависимостей"] B --> C["3. @PostConstruct
инициализация"] C --> D["4. Использование
рабочее состояние"] D --> E["5. @PreDestroy
уничтожение"]
СпособАннотация / интерфейсКогда вызывается
Конструктор—При создании бина
Инициализация@PostConstructПосле инъекции всех зависимостей
Уничтожение@PreDestroyПеред удалением бина из контекста
ИнициализацияInitializingBean.afterPropertiesSet()После инъекции свойств
УничтожениеDisposableBean.destroy()При закрытии контекста

Пример из проекта

@Component
public class LifecycleBean {

    private static final Logger log = LoggerFactory.getLogger(LifecycleBean.class);

    private final List<String> lifecycleEvents = new ArrayList<>();
    private int initCallCount = 0;
    private int destroyCallCount = 0;

    public LifecycleBean() {
        log.info("LifecycleBean constructor called");
        lifecycleEvents.add("CONSTRUCTOR");
    }

    @PostConstruct
    public void init() {
        initCallCount++;
        log.info("LifecycleBean @PostConstruct called (count: {})", initCallCount);
        lifecycleEvents.add("POST_CONSTRUCT");
        // Инициализация ресурсов, проверка конфигурации
    }

    @PreDestroy
    public void destroy() {
        destroyCallCount++;
        log.info("LifecycleBean @PreDestroy called (count: {})", destroyCallCount);
        lifecycleEvents.add("PRE_DESTROY");
        // Освобождение ресурсов
    }

    public List<String> getLifecycleEvents() { return lifecycleEvents; }
    public int getInitCallCount() { return initCallCount; }
}
Порядок вызовов: контейнер гарантирует, что @PostConstruct вызывается только после того, как все зависимости будут инъектированы. Для singleton-бинов @PostConstruct вызывается ровно один раз. @PreDestroy не вызывается для prototype-бинов — Spring не отслеживает их после создания.
В проекте: LifecycleBean.java с конструктором, @PostConstruct и @PreDestroy. Тест Section4_LifecycleTest.java проверяет порядок и количество вызовов. Метод lifecycleDemo() возвращает LifecycleResult.

5. @Configuration и @Bean

@Configuration — аннотация класса, который содержит методы @Bean. Каждый метод @Bean создаёт и возвращает объект, который регистрируется как бин в контейнере. Это способ декларативно определить бины для классов, которые вы не контролируете (библиотечные классы) или которым нужна сложная инициализация.

flowchart TD AppConfig["@Configuration AppConfig"] --> B1["@Bean defaultProduct()
→ Product"] AppConfig --> B2["@Bean electronicsCategory()
→ Category"] AppConfig --> B3["@Bean smartphoneProduct()
→ Product"] B3 -->|Зависит от| B2 AppConfig --> B4["@Bean @Primary primaryProduct()
→ Product"] IoC["IoC Container"] -->|Читает методы @Bean и регистрирует бины| AppConfig

Пример из проекта

@Configuration
public class AppConfig {

    // Бин продукта, созданный через @Bean
    @Bean(name = "defaultProduct")
    public Product defaultProduct() {
        return new Product("Default Product", "DEVICE", BigDecimal.valueOf(99.99));
    }

    // Бин категории для электроники
    @Bean(name = "electronicsCategory")
    public Category electronicsCategory() {
        return new Category("Electronics", "Electronic devices and accessories");
    }

    // Бин с зависимостью на другой бин из той же конфигурации
    @Bean(name = "smartphoneProduct")
    public Product smartphoneProduct(Category electronicsCategory) {
        Product product = new Product("Smartphone", "DEVICE", BigDecimal.valueOf(699.99));
        product.setCategory(electronicsCategory);
        return product;
    }

    // Primary бин — выбирается по умолчанию при инъекции по типу Product
    @Bean
    @Primary
    public Product primaryProduct() {
        return new Product("Primary Product", "DEFAULT", BigDecimal.valueOf(1.00));
    }
}

5.1 @Primary — бин по умолчанию

Когда в контейнере несколько бинов одного типа, @Primary указывает, какой бин использовать по умолчанию при инъекции по типу. 

@Bean
@Primary
public Product primaryProduct() {
    return new Product("Primary Product", "DEFAULT", BigDecimal.ONE);
}

// При инъекции по типу Product будет выбран primaryProduct
@Autowired
private Product product; // ← primaryProduct

// Альтернатива — получить конкретный бин по имени
@Autowired
@Qualifier("defaultProduct")
private Product defaultProduct; // ← defaultProduct

5.2 CGLIB-прокси для @Configuration

Важно: классы с @Configuration проксируются через CGLIB. Это гарантирует, что при вызове одного @Bean-метода из другого возвращается тот же singleton-экземпляр, а не создаётся новый. Классы с @Component такой прокси не имеют — межметодные вызовы создают новые экземпляры.
В проекте: AppConfig.java демонстрирует @Bean, @Primary и зависимости между бинами. Тест Section5_ConfigurationTest.java проверяет создание и инъекцию.

7. @Autowired и @Qualifier

@Autowired включает автоматическую инъекцию зависимостей. Когда в контейнере несколько бинов одного типа, @Qualifier помогает выбрать конкретный. Также можно получить бин по имени через context.getBean("name").

@Bean

8. @Value и Property Injection

@Value позволяет инъектировать значения из файлов конфигурации (application.yml / application.properties), переменных окружения и системных свойств. Environment предоставляет программный доступ к тем же источникам.

Синтаксис @Value

// Простое свойство
@Value("${app.name}")
private String appName;

// Значение по умолчанию (если свойство отсутствует)
@Value("${app.default-price:100.00}")
private BigDecimal defaultPrice;

// Булево значение
@Value("${app.feature-enabled:false}")
private boolean featureEnabled;

// SpEL (Spring Expression Language)
@Value("#{systemProperties['user.name']}")
private String userName;

9. ApplicationContextAware

Интерфейс ApplicationContextAware позволяет бину получить ссылку на ApplicationContext напрямую. Spring вызывает setApplicationContext() после создания бина и инъекции всех его зависимостей. 

@Component
public class ContextAwareBean implements ApplicationContextAware {

    private static final Logger log = LoggerFactory.getLogger(ContextAwareBean.class);

    private ApplicationContext applicationContext;

    @Override
    public void setApplicationContext(ApplicationContext applicationContext) {
        log.info("ApplicationContext injected into ContextAwareBean");
        this.applicationContext = applicationContext;
    }

    public ApplicationContext getApplicationContext() {
        return applicationContext;
    }

    public String getApplicationContextClassName() {
        return applicationContext.getClass().getSimpleName();
    }

    public int getBeanCount() {
        return applicationContext.getBeanDefinitionCount();
    }

    public boolean containsProductService() {
        return applicationContext.containsBeanDefinition("productService");
    }
}

Альтернативы (предпочтительные)

ПодходПреимущество
Прямая инъекция ApplicationContext в конструктор Нет привязки к интерфейсу Spring, можно сделать final
ObjectProvider<T> Получение нужного бина без зависимости от контекста
ApplicationEventPublisher Только публикация событий (узкая ответственность)
Рекомендация: не злоупотребляйте ApplicationContextAware. Чаще всего нужен именно конкретный бин, а не весь контейнер. Инжектите только то, что действительно нужно.
В проекте: ContextAwareBean.java. Тест Section9_ContextAwareTest.java проверяет, что контекст инжектирован и ссылается на тот же экземпляр, что и в тесте.

10. Профили (@Profile)

Профили позволяют регистрировать разные бины для разных окружений (dev, test, prod). Бин с @Profile("dev") создаётся только при активном профиле dev.

Определение профильных бинов

@Configuration
public class ProfileConfig {

    @Bean
    @Profile("dev")
    public Product devProduct() {
        return new Product("Dev Product", "DEV", BigDecimal.ZERO);
    }

    @Bean
    @Profile("prod")
    public Product prodProduct() {
        return new Product("Prod Product", "PROD", BigDecimal.valueOf(999.99));
    }

    @Bean
    @Profile("test")
    public Product testProduct() {
        return new Product("Test Product", "TEST", BigDecimal.ONE);
    }
}

11. События (Events)

Spring поддерживает паттерн publisher-subscriber для слабосвязанной коммуникации между компонентами. Контейнер сам публикует системные события (ContextRefreshedEvent, ContextClosedEvent), а вы можете создавать свои кастомные события.

sequenceDiagram participant Service as SpringContextDemoService participant Pub as EventPublisher participant AppCtx as ApplicationContext participant Listener as CustomEventListener Service->>Pub: publishEvent("Test event") Pub->>AppCtx: publishEvent(new CustomEvent(...)) AppCtx->>Listener: handleCustomEvent(event) Note right of Listener: @EventListener автоматически вызывается AppCtx->>Listener: handleContextRefreshed(event) Note right of Listener: Стандартное событие Spring

Создание кастомного события

public class CustomEvent extends ApplicationEvent {

    private final String message;
    private final long eventTimestamp;

    public CustomEvent(Object source, String message) {
        super(source);
        this.message = message;
        this.eventTimestamp = System.currentTimeMillis();
    }

    public String getMessage() { return message; }
    public long getEventTimestamp() { return eventTimestamp; }
}

Публикация события

@Component
public class EventPublisher {

    private final ApplicationEventPublisher eventPublisher;

    @Autowired
    public EventPublisher(ApplicationEventPublisher eventPublisher) {
        this.eventPublisher = eventPublisher;
    }

    public void publishEvent(String message) {
        CustomEvent event = new CustomEvent(this, message);
        eventPublisher.publishEvent(event);
    }
}

Обработка событий

@Component
public class CustomEventListener {

    private final List<String> capturedEvents = new ArrayList<>();
    private int customEventCount = 0;

    // Обработка кастомного события
    @EventListener
    public void handleCustomEvent(CustomEvent event) {
        capturedEvents.add("CUSTOM: " + event.getMessage());
        customEventCount++;
    }

    // Обработка системного события (контекст обновлён)
    @EventListener
    public void handleContextRefreshedEvent(ContextRefreshedEvent event) {
        capturedEvents.add("CONTEXT_REFRESHED");
    }

    public List<String> getCapturedEvents() { return capturedEvents; }
    public int getCustomEventCount() { return customEventCount; }
    public String getLastEventPayload() {
        return capturedEvents.isEmpty() ? null : capturedEvents.getLast();
    }
}

Стандартные события Spring

СобытиеКогда публикуется
ContextRefreshedEventПосле инициализации или обновления ApplicationContext
ContextStartedEventПосле вызова start()
ContextStoppedEventПосле вызова stop()

RequestHandledEventПосле обработки HTTP-запроса (web)
Асинхронные события: добавьте @Async к методу-обработчику, и он будет выполняться в отдельном потоке. Не забудьте @EnableAsync на конфигурации.
В проекте: CustomEvent.java, EventPublisher.java, CustomEventListener.java. Тест Section11_EventTest.java проверяет публикацию, приём и количество событий.

12. Условные бины (@Conditional)

Условные бины создаются только при выполнении определённого условия: наличие свойства, класса, другого бина или активного профиля.

Обзор условных аннотаций Spring Boot

АннотацияУсловиеПример
@ConditionalOnProperty Свойство имеет нужное значение @ConditionalOnProperty(name="feature.enabled", havingValue="true")
@ConditionalOnClass Класс есть в classpath @ConditionalOnClass(name="redis.clients.jedis.Jedis")
@ConditionalOnMissingBean Бин определённого типа ещё не создан @ConditionalOnMissingBean(DataSource.class)
@ConditionalOnBean Бин определённого типа существует @ConditionalOnBean(CacheManager.class)
@ConditionalOnExpression SpEL-выражение возвращает true @ConditionalOnExpression("${a} > ${b}")
@Profile Активный профиль совпадает @Profile("dev")

Пример из проекта

@Configuration
public class ConditionalConfig {

    // Бин создаётся, если feature.enabled=true.
    // matchIfMissing=true → создаётся даже если свойство отсутствует.
    @Bean
    @ConditionalOnProperty(
        name = "feature.enabled",
        havingValue = "true",
        matchIfMissing = true
    )
    public FeatureService enabledFeatureService() {
        return new FeatureService("ENABLED_FEATURE");
    }

    // Бин создаётся только в dev-профиле
    @Bean
    @Profile("dev")
    public DevToolsService devToolsService() {
        return new DevToolsService();
    }
}

16. Советы и лучшие практики

Предпочитайте конструкторную инъекцию

// ✅ Хорошо — явно, тестируемо, immutable
@Service
public class ProductService {
    private final ProductRepository repository;

    public ProductService(ProductRepository repository) {
        this.repository = repository;
    }
}

// ❌ Плохо — скрывает зависимости, не immutable
@Service
public class ProductService {
    @Autowired
    private ProductRepository repository;
}

Используйте final для неизменяемых зависимостей

private final ProductRepository repository; // final гарантирует инъекцию

Применяйте @Primary для выбора бина по умолчанию

@Bean
@Primary
public Product primaryProduct() {
    return new Product("Primary", "DEFAULT", BigDecimal.ONE);
}

Используйте @Profile для профилизации

@Bean
@Profile("dev")
public DataSource devDataSource() {
    // Конфигурация для разработки
}

Не смешивайте @Configuration и @Component

Не используйте @Component на классах, содержащих @Bean-методы. Без @Configuration (и CGLIB-прокси) межметодные вызовы будут создавать новые экземпляры вместо возврата singleton'а.

Разделяйте ответственность конфигураций

// Разные @Configuration классы для разных подсистем
@Configuration
public class DataSourceConfig { /* ... */ }

@Configuration
public class SecurityConfig { /* ... */ }

@Configuration
public class WebConfig { /* ... */ }

Проверяйте границы агрегатов (если применяете DDD)

Как и в Spring Data JDBC, в Spring IoC важно правильно определять границы бинов. Не делайте один God-класс-сервис на 3000 строк. Разделяйте на маленькие, сфокусированные сервисы с чёткими зависимостями — контейнер справится с этим.
