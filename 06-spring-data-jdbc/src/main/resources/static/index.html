1. Что такое Spring Data JDBC

Spring Data JDBC — это модуль Spring Data, предоставляющий доступ к реляционным базам данных без накладных расходов полноценного ORM (JPA/Hibernate). Он построен вокруг концепции агрегатов из Domain-Driven Design и обеспечивает простой, прозрачный и предсказуемый способ работы с данными.

Ключевая идея: каждый aggregate root загружается и сохраняется целиком, связи между агрегатами явные (через идентификаторы), а поведение при сохранении — максимально простое и близкое к SQL.

В отличие от JPA, здесь нет понятий persistence context, lazy loading, dirty checking или каскадных операций. Всё управляется вручную: вы читаете агрегат, меняете его и явно вызываете save().

graph TB APP[Application] --> Repo[Repository] Repo --> JDBC[Spring Data JDBC] JDBC --> JdbcTemplate[JdbcTemplate] JdbcTemplate --> DB[(Database)]

Проект использует H2 в качестве in-memory базы, схема создаётся через schema.sql, начальные данные загружаются через DemoDataInitializer.java.

2. Spring Data JDBC vs JPA

Основные отличия от традиционного подхода JPA/Hibernate. Эти различия возвращаются эндпоинтом /api/jdbc/differences и проверяются тестом Section1_DifferencesTest.java.

ТемаSpring Data JDBCJPA / Hibernate
Persistence ContextОтсутствуетEntityManager с контекстом
Lazy LoadingНет — всё загружается в пределах агрегатаЕсть, через прокси
Dirty CheckingНет — нужно явно вызывать save()Автоматическое на flush
СвязиДочерние коллекции внутри агрегата + AggregateReference@OneToMany, @ManyToOne, @ManyToMany
СохранениеВесь агрегат целиком (delete-insert для коллекций)Гибкое управление каскадами

Важно: Отсутствие lazy loading означает, что при загрузке агрегата загружаются и все его дочерние элементы. Это упрощает логику, но требует внимательного проектирования границ агрегатов.

3. Domain-Driven Design и агрегаты

Spring Data JDBC идеально сочетается с DDD-подходом. Основная единица работы — агрегат (aggregate), который представляет собой группу объектов, изменяемых как единое целое. Корень агрегата (aggregate root) — единственная сущность, имеющая репозиторий.

Агрегаты в проекте

classDiagram class Category { Long id String title List~CategoryProduct~ products addProduct() removeProduct() } class CategoryProduct { Long id String name String type BigDecimal price ProductDetails details AggregateReference~Tag,Long~ tag Long version } class ProductDetails { String manufacturer String sku } class Tag { Long id String name } class Customer { Long id String name String email List~CustomerOrder~ orders addOrder() } class CustomerOrder { Long id String description } Category "1" --> "*" CategoryProduct : contains CategoryProduct --> "1" ProductDetails : embedded CategoryProduct --> "1" Tag : references via AggregateReference Customer "1" --> "*" CustomerOrder : contains
  • Category — корень агрегата, владеет списком CategoryProduct.
  • Customer — корень агрегата, владеет списком CustomerOrder.
  • Tag — самостоятельный агрегат, на который ссылаются продукты через AggregateReference.

Репозитории созданы только для корней агрегатов: CategoryRepository, CustomerRepository, TagRepository. Дочерние сущности (CategoryProduct, CustomerOrder) не имеют собственных репозиториев и всегда сохраняются/удаляются вместе с родителем.

4. Маппинг сущностей и связи

Маппинг в Spring Data JDBC управляется аннотациями из модуля org.springframework.data.relational.core.mapping.

4.1 Aggregate root

Корневая сущность помечается @Table (можно опустить, если имя класса совпадает с именем таблицы), а поле идентификатора — @Id. 

@Table("categories")
public class Category {
  @Id
  private Long id;
  private String title;
  // ...
}

4.2 Дочерние коллекции (@MappedCollection)

Для хранения дочерних элементов внутри агрегата используется @MappedCollection. Она указывает, что дочерние записи хранятся в отдельной таблице и связаны с родителем через внешний ключ. 

@MappedCollection(idColumn = "category_id", keyColumn = "product_key")
private List<CategoryProduct> products = new ArrayList<>();
  • idColumn — имя колонки в дочерней таблице, содержащей ID родителя.
  • keyColumn — колонка, определяющая порядок элементов в коллекции (если используется List).

При сохранении агрегата все дочерние записи будут вставлены/обновлены/удалены автоматически (полная замена коллекции — см. секцию 5.1).

4.3 Value Object (@Embedded)

Value Object (объект-значение) не имеет собственного идентификатора и встраивается в таблицу владельца. В проекте ProductDetails встроен в CategoryProduct. 

@Embedded(onEmpty = Embedded.OnEmpty.USE_EMPTY)
private ProductDetails details;

Поля manufacturer и sku из ProductDetails будут сохранены в той же таблице category_product.

4.4 Ссылки между агрегатами (AggregateReference)

Для ссылки на другой агрегат используется AggregateReference<T, ID> — типобезопасная обёртка над идентификатором. Это явно показывает, что связь ведёт к другому корню. 

private AggregateReference<Tag, Long> tag;

В базе хранится просто числовой идентификатор, а при чтении ссылка остаётся ссылкой (объект Tag автоматически НЕ подгружается). Чтобы получить связанный объект, нужно выполнить отдельный запрос через его репозиторий.

4.5 Оптимистичная блокировка (@Version)

Для защиты от конкурентных обновлений используется аннотация @Version. Поле с этой аннотацией автоматически увеличивается при каждом сохранении, и если версия в базе изменилась — выбрасывается OptimisticLockingFailureException. 

@Version
private Long version;
Как работает: при UPDATE в условие WHERE добавляется текущая версия. Если строка не найдена (кто-то обновил раньше) — операция завершается с ошибкой, и приложение может повторить попытку или сообщить об этом пользователю.

4.6 Стратегия именования (NamingStrategy)

По умолчанию Spring Data JDBC использует NamingStrategy, которая преобразует имена из Java-стиля (camelCase) в SQL-стиль (snake_case):

JavaSQL
CategoryProduct (класс)таблица category_product
productName (поле)колонка product_name
customerId (поле)колонка customer_id

Явное указание имён через @Table("my_table") и @Column("my_column") имеет приоритет над стратегией именования. В проекте @Table указан на aggregate root'ах для наглядности.

5. Репозитории: иерархия ListCrudRepository

В Spring Data JDBC нет JpaRepository, вместо него используется ListCrudRepository (или CrudRepository), который предоставляет базовые CRUD-методы, возвращая List вместо Iterable.

graph BT MyRepo[CategoryRepository] -->|extends| LCR[ListCrudRepository - findAll → List] LCR -->|extends| CR[CrudRepository - save, findById, delete, ...] CR -->|extends| R[Repository - маркерный интерфейс]

ListCrudRepository включает методы: 

<S extends T> S save(S entity);
Optional<T> findById(ID id);
boolean existsById(ID id);
List<T> findAll();
List<T> findAllById(Iterable<ID> ids);
long count();
void deleteById(ID id);
void delete(T entity);
void deleteAllById(Iterable<? extends ID> ids);
void deleteAll(Iterable<? extends T> entities);
void deleteAll();

Этого достаточно для большинства сценариев. При необходимости можно добавлять derived query methods или аннотированные @Query методы.

5.1 Поведение save() — delete-insert для коллекций

Критически важно: при вызове save() для агрегата, Spring Data JDBC удаляет все дочерние записи коллекции из базы и вставляет их заново. Это гарантирует консистентность агрегата, но может быть неэффективно для больших коллекций.

Пример. Если у категории 100 продуктов и вы добавили 1 новый:

  1. Spring Data JDBC выполняет DELETE FROM category_product WHERE category_id = ?
  2. Затем 101 INSERT INTO category_product ...

Это осознанное проектное решение: агрегат — неделимая единица, и Spring Data JDBC не пытается вычислять diff. Если производительность критична, стоит пересмотреть границы агрегата (например, выделить продукт в отдельный агрегат).

5.2 Транзакции

Каждый вызов метода репозитория (save(), findById(), и т.д.) выполняется в отдельной транзакции. Если требуется объединить несколько операций в одну бизнес-транзакцию, используйте @Transactional на уровне сервиса: 

@Service
public class MyService {

    @Transactional
    public void transferProduct(Long fromCategoryId, Long toCategoryId, Long productId) {
        Category from = categoryRepository.findById(fromCategoryId).orElseThrow();
        Category to = categoryRepository.findById(toCategoryId).orElseThrow();

        CategoryProduct product = from.removeProduct(productId);
        to.addProduct(product);

        categoryRepository.save(from);  // ← обе операции
        categoryRepository.save(to);    // ← в одной транзакции
    }
}

6. Custom Queries

Spring Data JDBC поддерживает два способа определения собственных запросов: derived queries (по имени метода) и @Query с SQL.

Derived query example

List<Category> findByTitleContainingIgnoreCase(String titlePart);

Такой метод автоматически преобразуется в SQL: SELECT ... WHERE UPPER(title) LIKE UPPER('%' || ? || '%').

Явный SQL через @Query

@Query("""
    SELECT DISTINCT c.*
    FROM "categories" c
    JOIN "category_product" cp ON cp."category_id" = c."ID"
    WHERE cp."PRICE" >= :minPrice
    """)
List<Category> findCategoriesWithProductsAbovePrice(@Param("minPrice") BigDecimal minPrice);
Внимание: двойные кавычки вокруг идентификаторов в примере выше — особенность H2. Для PostgreSQL используйте двойные кавычки с другим регистром, для MySQL — обратные кавычки (`table`), а в большинстве случаев кавычки можно опустить, если имена таблиц/колонок не совпадают с зарезервированными словами.
GET /api/jdbc/custom-query?title=...
GET /api/jdbc/custom-query/min-price?minPrice=...

7. Lifecycle Events

Spring Data JDBC предоставляет события жизненного цикла сущностей, на которые можно подписаться с помощью @EventListener. В проекте реализован слушатель CategoryLifecycleListener.java, который собирает события в LifecycleEventsCollector.java.

События при сохранении и чтении

СобытиеМомент возникновения
BeforeConvertEventПеред преобразованием entity в структуру для сохранения
BeforeSaveEventПеред выполнением SQL INSERT/UPDATE
AfterSaveEventПосле успешного сохранения
AfterConvertEventПосле чтения из БД и создания entity

События при удалении

СобытиеМомент возникновения
BeforeDeleteEventПеред выполнением SQL DELETE
AfterDeleteEventПосле успешного удаления из БД
В демо-проекте слушатель подписан на события сохранения и чтения. События удаления (BeforeDeleteEvent, AfterDeleteEvent) также доступны, но не отслеживаются в LifecycleEventsCollector.

curl -X POST http://localhost:8890/api/jdbc/categories/1/products \

curl -X DELETE http://localhost:8890/api/jdbc/categories/1

Вывод

Spring Data JDBC предлагает простой и прозрачный путь работы с базами данных, особенно если вы придерживаетесь DDD и хотите явно контролировать границы агрегатов. Этот проект демонстрирует все основные возможности модуля: агрегаты, маппинг, ссылки между агрегатами, события, кастомные запросы и полноценный REST API.
