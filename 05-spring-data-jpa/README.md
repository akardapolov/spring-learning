# Spring Data JPA Demo

Демо‑проект на Spring Boot + Spring Data JPA + H2.

## Требования

- JDK 21+
- Maven 3.9+

Проверка:
```bash
java -version
mvn -v
```

## Запуск приложения локально

Из корня проекта:

### Вариант 1. Через Maven

```bash
mvn clean spring-boot:run
```

### Вариант 2. Через собранный JAR

```bash
mvn clean package
java -jar target/05-spring-data-jpa-1.0.0.jar
```

Приложение стартует на порту, указанном в `application.yaml`:

- `http://localhost:8889`

## Доступ к index.html

Файл `index.html` лежит в:

```text
src/main/resources/static/index.html
```

После запуска приложения откройте в браузере:

- `http://localhost:8889/`
  или
- `http://localhost:8889/index.html`

## Просмотр index.html без бэкенда (только вёрстка)

Если нужно просто открыть страницу как статический файл, без API и без запуска Spring Boot:

1. Найдите файл `src/main/resources/static/index.html`.
2. Откройте его в браузере через IDE или напрямую из файловой системы.

Адрес будет вида:

```text
file:///.../src/main/resources/static/index.html
```

## Консоль H2

Консоль встроенной БД доступна по адресу:

- `http://localhost:8889/h2-console`

Настройки подключения:

- JDBC URL: `jdbc:h2:mem:springdatajpa`
- User Name: `sa`
- Password: пусто

## Что демонстрирует проект

Проект показывает основные возможности Spring Data JPA:

- Spring Data Repository и иерархию интерфейсов
- Query Methods
- `@Query` и native query
- пагинацию и сортировку
- `Pageable`, `Page`, `Slice`
- projections
- specifications
- `@EntityGraph` в репозиториях
- работу с JPA‑моделями, сохранёнными максимально близко к модулю Hibernate

## Основные endpoint'ы

- `GET /api/demo/repository-hierarchy`
- `GET /api/demo/query-methods`
- `GET /api/demo/query-annotation`
- `GET /api/demo/pagination/page?page=0&size=3`
- `GET /api/demo/pagination/slice?page=0&size=2`
- `GET /api/demo/projections`
- `GET /api/demo/specifications?type=DEVICE&name=pho&minPrice=500`

### Сравнение основных подходов изменить функциональность нужного класса в Java

| Подход                                | Суть                                                                                   | Когда применять                                                                     | Плюсы                                                           | Минусы                                                                                                   | Используется ли в Spring                                                                                                          |
|---------------------------------------|----------------------------------------------------------------------------------------|-------------------------------------------------------------------------------------|-----------------------------------------------------------------|----------------------------------------------------------------------------------------------------------|-----------------------------------------------------------------------------------------------------------------------------------|
| Наследование                          | Создается подкласс, который переопределяет методы базового класса                      | Когда вы контролируете класс и хотите изменить/расширить поведение через `extends`  | Просто, понятно, встроено в язык                                | Жесткая связность, нельзя наследоваться от `final`, сложно комбинировать несколько независимых изменений | Да, но ограниченно. В пользовательском коде — часто. Внутри Spring как основной механизм расширения — реже, чем proxy/composition |
| Композиция                            | Новый класс содержит экземпляр другого класса и использует его                         | Когда нужно расширить поведение без наследования                                    | Гибче наследования, слабее связность, легче тестировать         | Нужно писать больше шаблонного кода                                                                      | Да, очень широко. Один из базовых подходов в Spring                                                                               |
| Делегирование                         | Объект передает выполнение другому объекту                                             | Когда нужно переадресовать поведение и иметь возможность подменять реализацию       | Просто, гибко, хорошо сочетается с интерфейсами                 | Часто много однотипного кода                                                                             | Да, очень широко                                                                                                                  |
| Wrapper / Обертка                     | Класс оборачивает другой объект и обычно повторяет его интерфейс                       | Когда нужно добавить поведение "снаружи", не меняя оригинал                         | Не меняет исходный класс, можно комбинировать                   | Много boilerplate, нужно прокидывать вызовы                                                              | Да. Часто встречается в инфраструктурных классах                                                                                  |
| Декоратор                             | Частный случай wrapper: добавляет поведение до/после вызова, сохраняя тот же интерфейс | Когда нужно наращивать поведение цепочкой                                           | Хорошо расширяется, можно комбинировать несколько декораторов   | Может усложнять цепочку вызовов                                                                          | Да. Идея используется часто, хотя не всегда явно называется decorator                                                             |
| Стратегия (Strategy)                  | Поведение выносится в отдельный объект/интерфейс и подставляется в основной класс      | Когда нужно менять алгоритм без изменения основного класса                          | Очень гибко, легко тестировать, хорошо сочетается с DI          | Нужно заранее спроектировать точки расширения                                                            | Да, очень широко. Один из ключевых стилей проектирования в Spring                                                                 |
| Шаблонный метод (Template Method)     | Базовый класс задает общий алгоритм, а части шагов переопределяются в подклассе        | Когда есть общий каркас логики и вариативные шаги                                   | Удобно для повторного использования общего алгоритма            | Основан на наследовании, менее гибок, чем композиция                                                     | Да. Часто встречается, например в классах вида `*Template` и базовых абстракциях                                                  |
| Dynamic Proxy (JDK Proxy)             | Во время выполнения создается прокси для интерфейса, перехватывающий вызовы методов    | Когда нужно перехватывать вызовы интерфейсных бинов: логирование, транзакции и т.п. | Без изменения исходного кода, удобно для cross-cutting concerns | Работает только с интерфейсами                                                                           | Да, очень широко. Один из основных механизмов Spring AOP                                                                          |
| CGLIB / ByteBuddy proxy               | Во время выполнения создается подкласс целевого класса и переопределяет методы         | Когда нужен proxy для класса, а не только интерфейса                                | Позволяет проксировать классы                                   | Не работает для `final class`, нельзя переопределять `final method`, есть runtime-сложность              | Да, широко. Spring использует class-based proxy, если нужно                                                                       |
| Proxy (паттерн в целом)               | Заместитель контролирует доступ к реальному объекту                                    | Когда нужен ленивый доступ, защита, кеш, удаленный вызов, логирование               | Универсальный паттерн, скрывает детали доступа                  | Может усложнять отладку и трассировку                                                                    | Да, очень широко. Одна из центральных идей Spring                                                                                 |
| AOP                                   | Вынесение сквозной логики в аспекты: транзакции, логирование, security, кеширование    | Когда нужно массово применять одинаковое поведение ко многим методам/бинам          | Убирает дублирование, удобно для инфраструктурной логики        | Может сделать поведение менее очевидным, сложнее дебажить                                                | Да, очень широко. Одна из ключевых возможностей Spring                                                                            |
| Interceptor / Filter / Advice         | Встраивание логики до/после основной обработки                                         | Когда нужно перехватывать запросы, вызовы, обработку сообщений                      | Хорошо подходит для middleware-подхода                          | Нужно понимать жизненный цикл и порядок цепочки                                                          | Да, очень широко: `HandlerInterceptor`, filters, advices и др.                                                                    |
| Listener / Observer                   | Подписчики реагируют на события объекта/системы                                        | Когда нужно добавлять реакцию на события без жесткой связи                          | Слабая связность, удобно расширять                              | Поведение становится распределенным по системе                                                           | Да, очень широко: `ApplicationListener`, события контекста и др.                                                                  |
| Лямбды / callbacks / functional style | Поведение передается как функция                                                       | Когда нужно точечно подставлять логику без новых классов                            | Кратко, удобно, уменьшает число классов                         | Не всегда подходит для сложного состояния и больших сценариев                                            | Да, используется, но скорее как общий стиль Java, а не как основной механизм Spring                                               |
| Bytecode instrumentation / Java Agent | Изменение байткода на лету или при загрузке класса                                     | Когда нужно глубокое вмешательство: мониторинг, профилирование, трассировка         | Очень мощно, можно менять поведение вообще без правки кода      | Сложно, низкоуровнево, трудно отлаживать                                                                 | В самом Spring — не основной способ. Но в экосистеме Java/Spring встречается через сторонние инструменты                          |
