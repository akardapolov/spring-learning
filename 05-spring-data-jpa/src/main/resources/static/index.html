1. Что такое Spring Data JPA

Spring Data JPA — это уровень абстракции поверх JPA, который упрощает работу с репозиториями, генерацией запросов, пагинацией, сортировкой, проекциями и динамической фильтрацией.

Важно:
JPA — это спецификация.
Hibernate — наиболее популярная реализация JPA.
Spring Data JPA — это не замена JPA и не замена Hibernate, а удобная интеграция и абстракция поверх них.
УровеньЧто даёт
JPAСтандартный API для ORM
HibernateРеализация JPA, SQL generation, dirty checking, persistence context
Spring Data JPARepository abstraction, query derivation, pagination, projections, specifications
graph TB APP[Application code] --> SD[Spring Data JPA] SD --> JPA[JPA API] JPA --> HIB[Hibernate] HIB --> JDBC[JDBC] JDBC --> DB[(H2 Database)]

Проблема: boilerplate-код для каждой сущности

Без Spring Data для каждой сущности приходится вручную писать однотипный DAO:

@Repository
public class ProductDao {

    @PersistenceContext
    private EntityManager em;

    public Product save(Product p) {
        em.persist(p);
        return p;
    }

    public Product findById(Long id) {
        return em.find(Product.class, id);
    }

    public List<Product> findAll() {
        return em.createQuery("select p from Product p", Product.class)
                 .getResultList();
    }

    public void delete(Product p) {
        em.remove(p);
    }

    // ...и так для КАЖДОЙ сущности писать одно и то же
}

Решение: один интерфейс вместо целого класса

Со Spring Data ты пишешь только интерфейс — реализацию Spring создаёт сам:

public interface ProductRepository extends JpaRepository<Product, Long> {
    // ВСЁ. Десятки методов уже доступны.
}
Без Spring Data JPAСо Spring Data JPA
Boilerplate DAO-код для каждой сущностиИнтерфейс — и реализация создаётся автоматически
Запросы писать вручнуюЧасть запросов выводится из имени метода
Пагинацию и сортировку настраивать вручнуюСтандартные Pageable, Page, Slice, Sort
DTO-выборки требуют отдельной ручной логикиМожно использовать projections
Динамические фильтры громоздкиМожно собирать Specification из условий
graph LR A[Service layer] --> B[Repository interface] B --> C[Spring Data proxy] C --> D[EntityManager] D --> E[Hibernate] E --> F[SQL]

2. Spring Data Repository: иерархия интерфейсов

Все готовые методы не свалены в один огромный интерфейс — они разложены по уровням. Каждый следующий уровень наследует предыдущий и добавляет своё.

Что даёт каждый уровень

ИнтерфейсЧто добавляет
RepositoryПустой маркер — просто говорит «я репозиторий»
CrudRepositorysave(), findById(), delete(), existsById(), count(), findAll() → Iterable
ListCrudRepositoryТе же CRUD-методы, но findAll() возвращает List вместо Iterable
PagingAndSortingRepositoryfindAll(Pageable), findAll(Sort) — пагинация и сортировка
ListPagingAndSortingRepositoryПагинация и сортировка с удобным List
JpaRepositoryОбъединяет обе ветки + JPA-специфика: flush(), saveAndFlush(), deleteAllInBatch(), getReferenceById()

Две ветки сходятся в одну

Есть CRUD-ветка и ветка пагинации. JpaRepository наследует обе — поэтому от него обычно и наследуются:

graph TD R["Repository — маркер"] CR["CrudRepository — save, findById, delete..."] LCR["ListCrudRepository — findAll() → List"] PSR["PagingAndSortingRepository — findAll(Pageable)"] LPSR["ListPagingAndSortingRepository — List + пагинация"] JR["JpaRepository — flush, batch, всё вместе"] PR["ProductRepository — твой интерфейс"] R --> CR CR --> LCR R --> PSR PSR --> LPSR LCR --> JR LPSR --> JR JR --> PR

Что ты получаешь, наследуясь от JpaRepository

// В сервисе — все эти методы уже есть, ни один не нужно писать:

productRepository.save(product);              // из CrudRepository
productRepository.findById(1L);               // из CrudRepository
productRepository.findAll();                  // из ListCrudRepository (→ List)
productRepository.deleteById(1L);             // из CrudRepository
productRepository.count();                    // из CrudRepository
productRepository.existsById(1L);             // из CrudRepository

productRepository.findAll(pageable);          // из PagingAndSortingRepository
productRepository.findAll(Sort.by("name"));   // из PagingAndSortingRepository

productRepository.flush();                    // из JpaRepository
productRepository.saveAndFlush(product);      // из JpaRepository
productRepository.deleteAllInBatch();         // из JpaRepository

Дополнительный интерфейс: JpaSpecificationExecutor

JpaSpecificationExecutor — не часть основной иерархии, а отдельный интерфейс для динамической фильтрации. Подключается при необходимости:

public interface ProductRepository
    extends JpaRepository<Product, Long>,          // CRUD + пагинация + JPA
            JpaSpecificationExecutor<Product> {     // + динамические фильтры
}

Как это работает под капотом

Ты пишешь только интерфейс — реализации нет. При запуске приложения Spring сканирует интерфейсы, наследующие Repository, и для каждого создаёт proxy-объект — динамическую реализацию, которая внутри делегирует вызовы в EntityManager:

graph LR A["ProductRepository
(интерфейс)"] --> B["Spring при старте
находит интерфейс"] B --> C["Создаёт proxy-объект
(динамическая реализация)"] C --> D["Proxy внутри вызывает
EntityManager"] D --> E["Hibernate
генерирует SQL"] E --> F[("База данных")]

Упрощённо, Spring создаёт за тебя что-то вроде:

// ЭТО Spring генерирует автоматически (ты этот код не пишешь):
public class ProductRepositoryProxy implements ProductRepository {

    private EntityManager em;

    public Product save(Product entity) {
        // внутри вызывает em.persist() или em.merge()
    }

    public Optional<Product> findById(Long id) {
        // внутри вызывает em.find(Product.class, id)
    }

    public List<Product> findAll() {
        // внутри создаёт JPQL: "select p from Product p"
    }

    public void deleteById(Long id) {
        // внутри вызывает em.find() + em.remove()
    }

    // ... все остальные методы из иерархии
}

3. Query Methods

Одна из самых известных возможностей Spring Data JPA — вывод запроса из имени метода репозитория.

Пример методаИдея
findByTypeOrderByName(String type)Найти продукты по типу и отсортировать по имени
findByCategory_Title(String title)Найти продукты по полю связанной сущности
findByPriceGreaterThanOrderByPriceDesc(BigDecimal price)Фильтр по числовому условию и сортировка
flowchart LR M[Method name] --> P[Spring Data parser] P --> Q[Derived query] Q --> JPA[JPA query execution] JPA --> DB[(Database)]

Примеры из проекта

List<Product> findByTypeOrderByName(String type);

List<Product> findByCategory_Title(String title);

List<Product> findByPriceGreaterThanOrderByPriceDesc(BigDecimal price);

Что удобно

  • не надо писать JPQL для простых сценариев;
  • можно идти по вложенным свойствам, например Category_Title;
  • можно добавлять сортировку прямо в имя метода.

Ограничения

Чем сложнее условие, тем менее читаемым становится имя метода. Для сложных сценариев лучше использовать @Query, projections или specifications.

4. @Query и native queries

Если query method уже не подходит по сложности или хочется явно контролировать текст запроса, используют @Query.

ПодходКогда использовать
@Query с JPQLКогда нужен читаемый объектно-ориентированный запрос
@Query(nativeQuery = true)Когда нужен SQL конкретной БД или уже есть готовый SQL

Примеры

@Query("select p from Product p where p.type = :type and p.price > :price order by p.name")
List<Product> findExpensiveByType(String type, BigDecimal price);

@Query(value = "select * from products where type = :type order by name", nativeQuery = true)
List<Product> findNativeByType(String type);
JPQL работает с entity и их полями, а не с таблицами и колонками.
native SQL работает напрямую с таблицами базы данных.
graph LR A["@Query JPQL"] --> B[Entity model] B --> C[Hibernate SQL generation] C --> DB[(DB)] D["@Query native SQL"] --> E[Raw SQL] E --> DB

Практика выбора

  • для обычной бизнес-логики сначала пробуют query methods;
  • если выражение становится слишком длинным — используют @Query;
  • native query берут, когда нужен именно SQL, который трудно или неудобно выразить через JPQL.

4.1. Сложные результаты из @Query

Простые запросы возвращают List<Product>. Но что если нужно вернуть вложенные объекты, агрегации, данные из нескольких таблиц?

Есть 6 подходов, каждый для своей ситуации:

ПодходВозвращаетКогда использовать
JOIN FETCH Entity с загруженными ассоциациями Нужен полный граф объекта с Category, Tags и т.д. одним SQL
select new DTO() Плоский DTO (record) Нужны поля из разных таблиц в одном плоском объекте
Nested projection Интерфейс с вложенными интерфейсами Read-only с вложенной структурой, без ручного маппинга
Tuple Tuple с доступом по алиасу Агрегации: GROUP BY, COUNT, AVG, MIN, MAX
Native + Object[] Массив объектов Сложный SQL с подзапросами, оконными функциями
Сборка в сервисе Map / List, собранные из нескольких запросов Сложный вложенный JSON-ответ для API

JOIN FETCH — загрузка связей одним запросом

@Query("select distinct p from Product p " +
       "join fetch p.category " +
       "left join fetch p.tags " +
       "where p.type = :type order by p.name")
List<Product> findWithCategoryAndTagsByType(String type);
Без fetch связи будут lazy — при сериализации в JSON получите LazyInitializationException. JOIN FETCH загружает всё одним SQL.

select new DTO() — плоский объект из JPQL

public record ProductSummaryDto(Long id, String name, BigDecimal price, String categoryTitle) {}

@Query("select new com.example.springdatajpa.dto.ProductSummaryDto(" +
       "p.id, p.name, p.price, p.category.title) " +
       "from Product p where p.type = :type order by p.name")
List<ProductSummaryDto> findSummariesByType(String type);
DTO должен быть плоским — нельзя передать коллекцию в конструктор. Зато можно собрать поля из нескольких связанных сущностей.

Вложенные интерфейсные проекции

public interface ProductFullView {
    Long getId();
    String getName();
    BigDecimal getPrice();

    CategoryView getCategory();       // вложенная проекция
    Set<TagView> getTags();           // коллекция вложенных проекций

    interface CategoryView {
        Long getId();
        String getTitle();
    }

    interface TagView {
        Long getId();
        String getName();
    }
}

List<ProductFullView> findProductFullViewByType(String type);
Вложенные проекции удобны, но под капотом могут генерировать N+1 запросов для коллекций. Для критичных мест лучше JOIN FETCH.

Tuple — аналитика и агрегации

@Query("select p.type as type, count(p) as productCount, " +
       "avg(p.price) as avgPrice, min(p.price) as minPrice, max(p.price) as maxPrice " +
       "from Product p group by p.type")
List<Tuple> findStatsByTypeTuple();

// В сервисе:
tuples.stream().map(t -> new TypeStatsDto(
    t.get("type", String.class),
    t.get("productCount", Long.class),
    BigDecimal.valueOf(t.get("avgPrice", Double.class)),
    t.get("minPrice", BigDecimal.class),
    t.get("maxPrice", BigDecimal.class)
));

Native SQL + ручной маппинг

@Query(value =
    "select p.id, p.name, p.price, p.type, c.title as category_title, " +
    "  (select count(*) from product_tags pt where pt.product_id = p.id) as tag_count " +
    "from products p join categories c on p.category_id = c.id " +
    "where p.type = :type order by p.name",
    nativeQuery = true)
List<Object[]> findProductReportNative(String type);

Сборка в сервисе

Иногда самый читаемый подход — несколько простых запросов, результаты которых собираются в Map/List в сервисном слое. Это особенно удобно для сложных вложенных JSON-ответов.

5. Пагинация и сортировка: Pageable, Page, Slice

Spring Data JPA умеет работать с пагинацией и сортировкой через стандартные абстракции.

ТипЧто содержитКогда удобен
PageableНомер страницы, размер, сортировкуВходной параметр для запроса
PageКонтент + total elements + total pagesКогда нужна полная информация о пагинации
SliceКонтент + информация, есть ли следующая частьКогда не нужен полный count(*)
SortОписание сортировкиОтдельно или внутри Pageable
flowchart LR P[Pageable] --> Q[Repository method] Q --> PG[Page or Slice] P --> S[Sort]

Примеры из проекта

Page<Product> findByPriceGreaterThan(BigDecimal price, Pageable pageable);

Slice<Product> findByType(String type, Pageable pageable);

Page

Page удобен, когда на UI нужно показать:
  • общее число элементов;
  • общее число страниц;
  • текущую страницу.

Slice

Slice легче: обычно не требует отдельного полного подсчёта всех строк и хорошо подходит для бесконечной прокрутки, «загрузить ещё» и похожих сценариев.

6. Projections

Проекции позволяют выбрать не всю entity, а только нужные поля. Это удобно для read-only сценариев и уменьшения объёма данных.

ВидИдея
Interface-based projectionSpring Data создаёт прокси под интерфейс с нужными геттерами
DTO projectionЗапрос возвращает готовый DTO

Пример интерфейсной проекции

public interface ProductView {
  Long getId();
  String getName();
  BigDecimal getPrice();
}
List<ProductView> findProjectedByTypeOrderByName(String type);
graph LR A[Repository method] --> B[Projection interface] B --> C[Only selected fields] C --> D[Smaller read model]
Когда это полезно:
  • списки для REST API;
  • таблицы на UI;
  • read-only сценарии;
  • когда не нужен полный граф сущности.
Проекции не заменяют entity-модель. Они хороши для чтения, но не для модификации доменного состояния.

7. Specifications

Specification используется для динамической сборки условий запроса. Это удобно, когда фильтры приходят от пользователя и их комбинация заранее неизвестна.

Пример спецификаций

public static Specification<Product> hasType(String type) {
  return (root, query, cb) -> cb.equal(root.get("type"), type);
}

public static Specification<Product> nameContains(String text) {
  return (root, query, cb) -> {
    String escaped = escapeLike(text.toLowerCase());
    return cb.like(cb.lower(root.get("name")), "%" + escaped + "%", '\\');
  };
}

public static Specification<Product> priceGreaterThan(BigDecimal price) {
  return (root, query, cb) -> cb.greaterThan(root.get("price"), price);
}

private static String escapeLike(String value) {
  return value
      .replace("\\", "\\\\")
      .replace("%", "\\%")
      .replace("_", "\\_");
}

Комбинация условий

List<Specification<Product>> specs = new ArrayList<>();

if (type != null && !type.isBlank()) {
  specs.add(ProductSpecification.hasType(type));
}
if (name != null && !name.isBlank()) {
  specs.add(ProductSpecification.nameContains(name));
}
if (minPrice != null) {
  specs.add(ProductSpecification.priceGreaterThan(minPrice));
}

Specification<Product> spec = Specification.allOf(specs);
flowchart TD A[Input filters] --> B[Specification builder] B --> C[Combined Specification] C --> D[JpaSpecificationExecutor] D --> E[Generated criteria query]
Для поддержки specifications репозиторий должен расширять JpaSpecificationExecutor<T>.
Плюсы:
  • удобно собирать фильтры по частям;
  • легче переиспользовать условия;
  • лучше читается, чем один огромный query method.
