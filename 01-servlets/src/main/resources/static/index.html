1. Что такое Servlet API и как проект устроен

Servlet API — это базовый Java API для серверной обработки HTTP-запросов. Он дает низкоуровневые механизмы для работы с запросом, ответом, сессиями, фильтрами и событиями жизненного цикла.

Уровень Что это Пример в проекте
Servlet Базовый интерфейс сервлета Используется не напрямую, а через HttpServlet
GenericServlet Абстрактная основа для сервлетов Промежуточный уровень в иерархии
HttpServlet Базовый класс для HTTP BasicHttpServlet, RequestApiServlet, ResponseApiServlet и др.
Spring MVC Более высокий уровень абстракции над HTTP DemoController
graph TB B[Browser / HTTP client] --> T[Embedded Tomcat] T --> F[Filters] F --> S[Servlets] F --> C[Spring MVC Controller] S --> R[JSON / text / redirect] C --> R
Идея проекта: показать сразу два уровня:
  • чистый Servlet API через @WebServlet, @WebFilter, @WebListener;
  • Spring MVC через @RestController для обзорных endpoint'ов /api/demo/*.

2. Запуск приложения и регистрация компонентов

Приложение работает на Spring Boot, но servlet-компоненты регистрируются не как Spring beans, а через стандартные jakarta.servlet-аннотации.

@SpringBootApplication
@ServletComponentScan
public class ServletsApplication {
  public static void main(String[] args) {
    SpringApplication.run(ServletsApplication.class, args);
  }
}
Аннотация Назначение Пример
@ServletComponentScan Включает поиск servlet-компонентов ServletsApplication
@WebServlet Регистрирует сервлет BasicHttpServlet, RequestApiServlet
@WebFilter Регистрирует фильтр LoggingFilter, TimingFilter
@WebListener Регистрирует listener ApplicationServletContextListener, DemoDataInitializer
flowchart LR A["Spring Boot start"] --> B["ServletComponentScan"] B --> C["scan WebServlet"] B --> D["scan WebFilter"] B --> E["scan WebListener"] C --> F["Servlets registered"] D --> G["Filters registered"] E --> H["Listeners registered"]
У нескольких сервлетов задан loadOnStartup. Это означает, что контейнер может инициализировать их уже при старте приложения, а не ждать первый запрос.

3. Жизненный цикл сервлета и многопоточность

Сервлет обычно создается контейнером один раз и затем используется для обработки множества запросов.

Этап Метод Смысл
Инициализация init() Контейнер создает и подготавливает сервлет
Обработка service() Распределяет вызов в doGet(), doPost(), doPut() и т.д.
Завершение destroy() Контейнер уничтожает сервлет при остановке приложения
sequenceDiagram participant Client participant Container participant Servlet Container->>Servlet: init() Client->>Container: HTTP request Container->>Servlet: service(req, resp) Servlet->>Servlet: doGet()/doPost()/doPut()/doDelete() Servlet-->>Client: HTTP response Container->>Servlet: destroy()

HTTP-методы в проекте

Метод Где показан Что демонстрирует
GET Практически все сервлеты Чтение данных
POST BasicHttpServlet, RequestApiServlet, SessionCookieServlet, ErrorHandlingServlet Создание ресурса / чтение body / установка session attributes
PUT BasicHttpServlet Обновление ресурса
DELETE BasicHttpServlet, SessionCookieServlet Удаление ресурса / удаление session attribute
OPTIONS BasicHttpServlet Разрешенные методы и заголовки
HEAD BasicHttpServlet Ответ без body
Важно про потоки: один экземпляр сервлета может обрабатывать много запросов одновременно. Поэтому нельзя хранить данные конкретного запроса в полях сервлета.
В проекте это учтено правильно:
  • данные запроса живут в локальных переменных методов;
  • общие демо-данные лежат в ServletContext;
  • для них используются потокобезопасные структуры: ConcurrentHashMap и AtomicLong.

  • DemoDataInitializer.java

4. HttpServletRequest и области данных

HttpServletRequest содержит все, что пришло от клиента: метод, URI, query-параметры, заголовки, body, cookies, атрибуты, сессию.

Что показывает RequestApiServlet

Метод API Что делает Где используется
getMethod() Возвращает HTTP-метод RequestApiServlet#doGet
getRequestURI() URI запроса RequestInfo
getContextPath() Контекст приложения RequestInfo
getServletPath() Путь к сервлету RequestInfo
getParameterMap() Все query/form параметры RequestApiServlet#doGet
getHeaderNames(), getHeader() Чтение заголовков RequestApiServlet#doGet
getReader() Чтение body как текста RequestApiServlet#doPost
getAttribute(), setAttribute() Работа с атрибутами запроса listeners и forward-сценарии 
ApiModels.RequestInfo requestInfo = new ApiModels.RequestInfo(
    req.getMethod(),
    req.getRequestURI(),
    req.getContextPath(),
    req.getServletPath(),
    req.getQueryString(),
    parameterMap,
    headers,
    req.getRemoteAddr(),
    req.getRemotePort(),
    req.getProtocol(),
    req.getScheme()
);

Области хранения данных

Scope Где живет Пример в проекте
Request scope Только в рамках одного HTTP-запроса requestStartTime, forwardedFrom, customAttr
Session scope Между запросами одного клиента атрибуты в SessionCookieServlet
Application scope На все приложение demoProducts, appStartTime
flowchart TB A["HTTP request"] --> B["ApplicationServletRequestListener"] B --> C["requestStartTime attribute added"] C --> D["Servlet or Controller"] D --> E["RequestApiServlet reads params, headers and attributes"]
Интересный момент: в forward-сценариях атрибуты запроса сохраняются. Именно поэтому в /api/forward/with-attributes можно передать атрибуты и затем увидеть их в /api/request.

5. HttpServletResponse и формат API-ответов

HttpServletResponse отвечает за то, что именно получит клиент: статус, заголовки, тип контента, кодировку, redirect и тело ответа.

Что показывает ResponseApiServlet

Метод API Смысл Где показан
setStatus(code) Установить HTTP статус /api/response/status/{code}
setHeader(name, value) Добавить заголовки /api/response/headers
setContentType(type) Тип тела ответа /api/response/content
setCharacterEncoding("UTF-8") Кодировка ответа /api/response/encoding
sendRedirect(to) Клиентский redirect /api/response/redirect
setBufferSize(), resetBuffer() Управление буфером /api/response/buffer
getWriter() Запись текстового тела Во всех JSON-ответах

Единый формат ответов

В проекте есть отдельные record-модели для API-ответов в файле ApiModels.java.

Модель Назначение
HttpResponse Успешный ответ: статус, сообщение, данные, timestamp
ErrorResponse Ошибка: статус, тип ошибки, сообщение, path, timestamp
CookieInfo, SessionInfo, RequestInfo DTO для демонстрации Servlet API

Пример успешного ответа

{
  "status": 200,
  "message": "Products retrieved successfully",
  "data": {
    "1": {
      "id": 1,
      "name": "Smartphone X"
    }
  },
  "timestamp": "2026-03-25T12:00:00Z"
}

Пример ответа с ошибкой

{
  "status": 404,
  "error": "Not Found",
  "message": "Product not found",
  "path": "/api/basic/products/999",
  "timestamp": 1710000000000
}
Практический смысл: единый формат ответа делает API предсказуемым и удобным для тестов и frontend-клиентов.

Как устроены данные

В проекте нет БД — данные создаются в памяти приложения через DemoDataInitializer, который реализует ServletContextListener.

Данные Где хранятся Зачем
Товары ServletContext["demoProducts"] Общий in-memory store для CRUD
Категории ServletContext["demoCategories"] Справочные данные
Теги ServletContext["demoTags"] Справочные данные
Генератор id ServletContext["productIdGenerator"] AtomicLong для создания новых id
flowchart LR A[Application start] --> B[DemoDataInitializer] B --> C[ServletContext] C --> D[demoProducts: ConcurrentHashMap] C --> E[productIdGenerator: AtomicLong] F[BasicHttpServlet] --> C

Модель данных

  • Product.java — товар
  • Category.java — категория
  • Tag.java — тег
Полезная деталь: Product — это Java record. В его конструкторе есть нормализация: если tags == null, то используется emptySet(). Это делает JSON-модель стабильнее.

Как выглядит CRUD по методам

doDelete() Удалить товар по id
doOptions() Показать разрешенные методы и заголовки

Это хороший учебный пример того, как REST-подобный API можно реализовать без Spring MVC, только средствами HttpServlet.

7. Session и Cookie API

HTTP по своей природе stateless, поэтому для хранения состояния между запросами используется HttpSession. Cookies помогают клиенту идентифицироваться между запросами.

GET /api/session
GET /api/session/create
GET /api/session/invalidate
GET /api/session/attributes
GET /api/session/cookies
POST /api/session
DELETE /api/session?key=name
API Смысл Где показан
req.getSession(true) Создать/получить сессию /api/session/create, POST /api/session
req.getSession(false) Получить существующую без создания /api/session, /invalidate
session.setAttribute() Сохранить данные в сессии handleSetAttribute
session.invalidate() Уничтожить сессию /api/session/invalidate
resp.addCookie(cookie) Отправить cookie клиенту handleCreateSession
req.getCookies() Прочитать cookies /api/session, /cookies
sequenceDiagram participant Client participant Servlet as SessionCookieServlet participant Session as HttpSession Client->>Servlet: GET /api/session/create Servlet->>Session: req.getSession(true) Servlet->>Session: setAttribute(createdAt, userAgent) Servlet-->>Client: Set-Cookie + JSON Client->>Servlet: GET /api/session Servlet->>Session: req.getSession(false) Servlet-->>Client: SessionInfo + CookieInfo Client->>Servlet: GET /api/session/invalidate Servlet->>Session: invalidate() Servlet-->>Client: remove cookie + JSON
В проекте есть отдельные DTO:
  • ApiModels.SessionInfo
  • ApiModels.CookieInfo
Это делает демонстрацию наглядной: сессия и cookies возвращаются в удобном JSON-виде.
Связь с listeners: создание и уничтожение сессий отслеживает ApplicationHttpSessionListener, который считает активные сессии через AtomicInteger.

8. Filters

Фильтр — это промежуточный слой, который может выполнить код до и после обработки запроса сервлетом или контроллером.

flowchart LR A[Client request] --> B[LoggingFilter] B --> C[TimingFilter] C --> D[Servlet / Controller] D --> C C --> B B --> E[Response]
Фильтр URL pattern Что делает
LoggingFilter /* Логирует вход и выход запроса, считает длительность
TimingFilter /api/* Отмечает медленные запросы

Ключевая идея Filter API

public void doFilter(ServletRequest request,
                     ServletResponse response,
                     FilterChain chain) {
    // код до сервлета
    chain.doFilter(request, response);
    // код после сервлета
}
LoggingFilter также показывает использование @WebInitParam:
  • logHeaders=true
  • logParams=false
В текущем демо эти init-параметры читаются при init(), а сам фильтр логирует метод, URI и длительность запроса.
Фильтры полезны для сквозной логики: логирование, CORS, аутентификация, метрики, сжатие, кодировка, аудит.

9. Listeners

Listeners реагируют на события жизненного цикла приложения, сессий и запросов.

Listener Интерфейс Что делает в проекте
ApplicationServletContextListener ServletContextListener Сохраняет appStartTime и логирует старт/остановку приложения
DemoDataInitializer ServletContextListener Инициализирует демо-данные в ServletContext
ApplicationHttpSessionListener HttpSessionListener Считает количество активных сессий
ApplicationServletRequestListener ServletRequestListener Сохраняет время начала запроса
flowchart TB A[Application start] --> B[ServletContextListeners] B --> C[appStartTime] B --> D[demoProducts/demoCategories/demoTags] E[New HTTP request] --> F[ServletRequestListener] F --> G[requestStartTime] H[New session] --> I[HttpSessionListener] I --> J[activeSessions++]
Точность по коду: в ApplicationServletRequestListener длительность запроса вычисляется, но вывод в консоль сейчас закомментирован. То есть listener уже собирает данные, но не шумит логами.
DemoController использует ApplicationHttpSessionListener.getActiveSessionCount(), чтобы отдать число активных сессий в /api/demo/listener-api.

10. Forward vs Redirect

Обе техники переводят пользователя в другой обработчик, но механика у них принципиально разная.

Критерий Forward Redirect
Где происходит На сервере На клиенте
URL в браузере Не меняется Меняется
Атрибуты request Сохраняются Теряются, потому что идет новый запрос
Количество HTTP-запросов Один Минимум два
Инструмент RequestDispatcher.forward() sendRedirect() или статус + Location
sequenceDiagram participant Client participant Server Note over Client,Server: Forward Client->>Server: GET /api/forward/with-attributes Server->>Server: request.setAttribute(...) Server->>Server: forward("/api/request") Server-->>Client: one final response Note over Client,Server: Redirect Client->>Server: GET /api/redirect/temporary Server-->>Client: 302 + Location: /api/demo Client->>Server: GET /api/demo Server-->>Client: new response

Что показывает код

  • /api/forward/to-welcome — серверный forward
  • /api/forward/with-attributes — forward с атрибутами запроса
  • /api/forward/chain — цепочка forward
  • /api/forward/loop — защита от бесконечного цикла
  • /api/redirect/temporary — 302
  • /api/redirect/permanent — 301
  • /api/redirect/see-other — 303
В ForwardRedirectServlet redirect показан вручную через установку статуса и заголовка Location, а в ResponseApiServlet есть отдельная демонстрация через resp.sendRedirect(to).

11. Обработка ошибок

ErrorHandlingServlet показывает два подхода:

  • явно вернуть JSON-ошибку со статусом;
  • намеренно выбросить исключение, чтобы увидеть поведение контейнера/фреймворка.
GET /api/error
GET /api/error/bad-request
GET /api/error/not-found
GET /api/error/forbidden
GET /api/error/server-error
GET /api/error/runtime
GET /api/error/illegal-argument
GET /api/error/null-pointer
POST /api/error
Сценарий Что происходит
/bad-request Возвращается 400 с ErrorResponse
/not-found Возвращается 404 с ErrorResponse
/forbidden Возвращается 403
/server-error Возвращается 500
/runtime, /illegal-argument, /null-pointer Сервлет намеренно бросает исключения
POST /api/error Проверяется Content-Type и обязательное поле name

Ключевая идея

private void sendErrorResponse(HttpServletResponse resp,
                               int status,
                               String error,
                               String message,
                               String path) throws IOException {
    ApiModels.ErrorResponse errorResponse =
        new ApiModels.ErrorResponse(status, error, message, path);
    resp.setStatus(status);
    sendJsonResponse(resp, errorResponse);
}
Практический смысл: вместо HTML error page клиент получает предсказуемый JSON. Это важно для REST API, автотестов и frontend-приложений.

Финальное резюме

Этот проект показывает базовые механизмы Servlet API не в отрыве от практики, а через конкретный код:

  • Servlets — обработка HTTP-методов и CRUD;
  • Request/Response — доступ к данным запроса и формирование ответа;
  • Session/Cookie — хранение состояния между запросами;
  • Filters — сквозная обработка;
  • Listeners — реакция на события жизненного цикла;
  • Forward/Redirect — два разных способа перенаправления;
  • Error handling — статус-коды, JSON-ошибки и исключения;
  • Spring MVC — более высокий уровень поверх той же HTTP-инфраструктуры.

Главное, что полезно вынести: Spring Web не отменяет Servlet API, а строится поверх него. Этот проект помогает увидеть основу.
