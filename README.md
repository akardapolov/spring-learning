# [Spring Learning Project](https://akardapolov.github.io/spring-learning)

Учебный проект, разбитый на модули для последовательного изучения экосистемы Spring и связанных с ней технологий.

Каждый модуль представляет собой отдельный проект с собственной документацией и примерами кода.

🌐 **Проект на GitHub Pages:** [spring-learning](https://akardapolov.github.io/spring-learning)

## Содержание курса

### 1. Основы Web (Servlets & Tomcat)
Фундамент Java Web, на котором базируется Spring MVC.
* Жизненный цикл HTTP запроса и ответа
* Servlet API: init, service, destroy
* Фильтры (Filters) и слушатели (Listeners)
* Управление сессиями (Session management)
* Глоссарий терминов
* Структура проекта
* Endpoint'ы (URL маппинги)
* 🔗 [Документация модуля](./01-servlets/src/main/resources/static/index.html)

### 2. Spring Context (Core)
Базовые концепции Spring Framework без магии автоконфигураций.
* Inversion of Control (IoC) и Dependency Injection (DI)
* Жизненный цикл Bean'а и Scopes (Singleton, Prototype и др.)
* Способы конфигурации: XML, Java Config, Annotations
* ApplicationContext vs BeanFactory
* Основы AOP (Aspect-Oriented Programming) и проксирование
* Глоссарий терминов
* Структура проекта
* 🔗 [Документация модуля](./02-spring-context/src/main/resources/static/index.html)

### 3. Spring Boot
Магия автоконфигурации и быстрый старт приложений.
* Что такое Spring Boot и зачем он нужен
* Стартеры (Starters) и управление зависимостями
* Механизм Auto-configuration (@Conditional)
* Конфигурация через `application.yaml` / `application.properties`
* Spring Boot Actuator: метрики и мониторинг
* Глоссарий терминов
* Структура проекта
* Endpoint'ы (Actuator)
* 🔗 [Документация модуля](./03-spring-boot/src/main/resources/static/index.html)

### 4. Hibernate
Глубокое погружение в ORM.
* Что такое Hibernate и зачем он нужен
* Жизненный цикл сущности (Entity Lifecycle)
* Связи между сущностями (`OneToMany`, `ManyToOne` и др.)
* Стратегии загрузки: `LAZY` vs `EAGER`
* Проблема `N+1` и способы её решения (`Join Fetch`, `Entity Graph`)
* Кэширование (`L1`, `L2`, `Query Cache`)
* Advanced scenarios (`Batching`, `Optimistic Locking`)
* Глоссарий терминов
* Структура проекта
* Endpoint'ы API
* 🔗 [Документация модуля](./04-hibernate/src/main/resources/static/index.html)

### 5. Spring Data JPA
Абстракция над JPA и упрощение работы с базой данных.
* Spring Data Repository: иерархия интерфейсов
* Query Methods: генерация запросов по имени метода
* Использование `@Query` и нативных запросов
* Пагинация и сортировка (`Pageable`, `Slice`)
* Projections (проекции) для оптимизации выборок
* Specifications (спецификации) для динамической фильтрации
* Глоссарий терминов
* Структура проекта
* Endpoint'ы API
* 🔗 [Документация модуля](./05-spring-data-jpa/src/main/resources/static/index.html)

### 6. Spring Data JDBC
Легковесная альтернатива JPA без "магии" сессий.
* Отличия Spring Data JDBC от JPA
* Подход Domain-Driven Design (DDD) и агрегаты
* Настройка маппинга и связей
* Написание кастомных запросов
* События жизненного цикла (Lifecycle Events)
* Глоссарий терминов
* Структура проекта
* Endpoint'ы API
* 🔗 [Документация модуля](./06-spring-data-jdbc/src/main/resources/static/index.html)

### 7. Spring Web (REST API)
Создание современных и правильных RESTful веб-сервисов.
* Архитектура REST и уровни зрелости
* Контроллеры: `@RestController`, `@RequestMapping`
* Сериализация / десериализация (`Jackson JSON`)
* Глобальная обработка ошибок (`@ControllerAdvice`)

* Структура проекта
* Endpoint'ы API
* 🔗 [Документация модуля](./07-spring-rest/src/main/resources/static/index.html)

### 8. Spring Security
Обеспечение безопасности веб-приложений и API.
* Архитектура Security Filter Chain
* Аутентификация vs авторизация
* Управление пользователями (`UserDetailsService`)
* Хэширование паролей (`PasswordEncoder`)
* JWT (`JSON Web Tokens`) и Stateless security
* Method Security (`@PreAuthorize`, `@Secured`)
* Защита от атак: CORS и CSRF
* Глоссарий терминов
* Структура проекта
* Endpoint'ы API
* 🔗 [Документация модуля](./08-spring-security/src/main/resources/static/index.html)

## Структура проекта

Проект организован как **Maven multi-module project**.

```text
spring/
├─ pom.xml
├─ README.md
├─ .gitignore
├─ 01-servlets/
├─ 02-spring-context/
├─ 03-spring-boot/
├─ 04-hibernate/
├─ 05-spring-data-jpa/
├─ 06-spring-data-jdbc/
├─ 07-spring-rest/
└─ 08-spring-security/
```

## Как использовать

В каждой папке находится отдельный модуль.  
Для изучения конкретной темы:

1. Перейдите в папку нужного модуля.
2. Откройте `index.html` для изучения теории и устройства модуля.
3. При наличии исполняемого приложения запустите модуль и протестируйте доступные endpoint'ы.

## Сборка и запуск

### Компиляция и сборка всего проекта

Из корневой папки проекта:

```bash
mvn clean compile
```

```bash
mvn clean install
```

Эта команда соберёт все модули.

### Сборка конкретного модуля

Пример для Servlets:

```bash
mvn -pl 01-servlets clean install
```

### Сборка и запуск конкретного модуля из корня проекта

#### 1. Основы Web (Servlets & Tomcat)
```bash
mvn -pl 01-servlets spring-boot:run
```

#### 2. Spring Context
```bash
mvn -pl 02-spring-context spring-boot:run
```

#### 3. Spring Boot
```bash
mvn -pl 03-spring-boot spring-boot:run
```

#### 4. Hibernate
```bash
mvn -pl 04-hibernate spring-boot:run
```

#### 5. Spring Data JPA
```bash
mvn -pl 05-spring-data-jpa spring-boot:run
```

#### 6. Spring Data JDBC
```bash
mvn -pl 06-spring-data-jdbc spring-boot:run
```

#### 7. Spring Web (REST API)
```bash
mvn -pl 07-spring-rest spring-boot:run
```

#### 8. Spring Security
```bash
mvn -pl 08-spring-security spring-boot:run
```

### Запуск из папки модуля

Можно перейти в директорию модуля и запускать его отдельно:

```bash
cd 04-hibernate
mvn spring-boot:run
```

### ООП и паттерны в Spring

| Подход/Паттерн                        | Суть                                                                                   | Когда применять                                                                     | Плюсы                                                           | Минусы                                                                                                   | Используется ли в Spring                                                                                                          |
|---------------------------------------|----------------------------------------------------------------------------------------|-------------------------------------------------------------------------------------|-----------------------------------------------------------------|----------------------------------------------------------------------------------------------------------|-----------------------------------------------------------------------------------------------------------------------------------|
| Наследование                          | Создается подкласс, который переопределяет методы базового класса                      | Когда вы контролируете класс и хотите изменить/расширить поведение через `extends`  | Просто, понятно, встроено в язык                                | Жесткая связность, нельзя наследоваться от `final`, сложно комбинировать несколько независимых изменений | Да, но ограниченно. В пользовательском коде — часто. Внутри Spring как основной механизм расширения — реже, чем proxy/composition |
| Композиция                            | Новый класс содержит экземпляр другого класса и использует его                         | Когда нужно расширить поведение без наследования                                    | Гибче наследования, слабее связность, легче тестировать         | Нужно писать больше шаблонного кода                                                                      | Да, очень широко. Один из базовых подходов в Spring                                                                               |
| Делегирование                         | Объект передает выполнение другому объекту                                             | Когда нужно переадресовать поведение и иметь возможность подменять реализацию       | Просто, гибко, хорошо сочетается с интерфейсами                 | Часто много однотипного кода                                                                             | Да, очень широко                                                                                                                  |
| Wrapper / Обертка                     | Класс оборачивает другой объект и обычно повторяет его интерфейс                       | Когда нужно добавить поведение "снаружи", не меняя оригинал                         | Не меняет исходный класс, можно комбинировать                   | Много boilerplate, нужно прокидывать вызовы                                                              | Да. Часто встречается в инфраструктурных классах                                                                                  |
| Декоратор                             | Частный случай wrapper: добавляет поведение до/после вызова, сохраняя тот же интерфейс | Когда нужно наращивать поведение цепочкой                                           | Хорошо расширяется, можно комбинировать несколько декораторов   | Может усложнять цепочку вызовов                                                                          | Да. Идея используется часто, хотя не всегда явно называется decorator                                                             |
| Стратегия (Strategy)                  | Поведение выносится в отдельный объект/интерфейс и подставляется в основной класс      | Когда нужно менять алгоритм без изменения основного класса                          | Очень гибко, легко тестировать, хорошо сочетается с DI          | Нужно заранее спроектировать точки расширения                                                            | Да, очень широко. Один из ключевых стилей проектирования в Spring                                                                 |
| Шаблонный метод (Template Method)     | Базовый класс задает общий алгоритм, а части шагов переопределяются в подклассе        | Когда есть общий каркас логики и вариативные шаги                                   | Удобно для повторного использования общего алгоритма            | Основан на наследовании, менее гибок, чем композиция                                                     | Да. Часто встречается, например в классах вида `*Template` и базовых абстракциях                                                  |
| Dynamic Proxy (JDK Proxy)             | Во время выполнения создается прокси для интерфейса, перехватывающий вызовы методов    | Когда нужно перехватывать вызовы интерфейсных бинов: логирование, транзакции и т.п. | Без изменения исходного кода, удобно для cross-cutting concerns | Работает только с интерфейсами                                                                           | Да, очень широко. Один из основных механизмов Spring AOP                                                                          |
| CGLIB / ByteBuddy proxy               | Во время выполнения создается подкласс целевого класса и переопределяет методы         | Когда нужен proxy для класса, а не только интерфейса                                | Позволяет проксировать классы                                   | Не работает для `final class`, нельзя переопределять `final method`, есть runtime-сложность              | Да, широко. Spring использует class-based proxy, если нужно                                                                       |
| Proxy (паттерн в целом)               | Заместитель контролирует доступ к реальному объекту                                    | Когда нужен ленивый доступ, защита, кеш, удаленный вызов, логирование               | Универсальный паттерн, скрывает детали доступа                  | Может усложнять отладку и трассировку                                                                    | Да, очень широко. Одна из центральных идей Spring                                                                                 |
| AOP                                   | Вынесение сквозной логики в аспекты: транзакции, логирование, security, кеширование    | Когда нужно массово применять одинаковое поведение ко многим методам/бинам          | Убирает дублирование, удобно для инфраструктурной логики        | Может сделать поведение менее очевидным, сложнее дебажить                                                | Да, очень широко. Одна из ключевых возможностей Spring                                                                            |
| Interceptor / Filter / Advice         | Встраивание логики до/после основной обработки                                         | Когда нужно перехватывать запросы, вызовы, обработку сообщений                      | Хорошо подходит для middleware-подхода                          | Нужно понимать жизненный цикл и порядок цепочки                                                          | Да, очень широко: `HandlerInterceptor`, filters, advices и др.                                                                    |
| Listener / Observer                   | Подписчики реагируют на события объекта/системы                                        | Когда нужно добавлять реакцию на события без жесткой связи                          | Слабая связность, удобно расширять                              | Поведение становится распределенным по системе                                                           | Да, очень широко: `ApplicationListener`, события контекста и др.                                                                  |
| Лямбды / callbacks / functional style | Поведение передается как функция                                                       | Когда нужно точечно подставлять логику без новых классов                            | Кратко, удобно, уменьшает число классов                         | Не всегда подходит для сложного состояния и больших сценариев                                            | Да, используется, но скорее как общий стиль Java, а не как основной механизм Spring                                               |
| Bytecode instrumentation / Java Agent | Изменение байткода на лету или при загрузке класса                                     | Когда нужно глубокое вмешательство: мониторинг, профилирование, трассировка         | Очень мощно, можно менять поведение вообще без правки кода      | Сложно, низкоуровнево, трудно отлаживать                                                                 | В самом Spring — не основной способ. Но в экосистеме Java/Spring встречается через сторонние инструменты                          |
